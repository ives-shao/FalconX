# FalconX v1 状态机规范

## 1. 目标

统一一期关键业务对象的状态定义、允许迁移方向和 owner。

## 2. 用户状态

owner：

- `identity-service`

状态：

- `PENDING_DEPOSIT`
- `ACTIVE`
- `FROZEN`
- `BANNED`

允许迁移：

- `PENDING_DEPOSIT -> ACTIVE`
- `ACTIVE -> FROZEN`
- `FROZEN -> ACTIVE`
- `ACTIVE -> BANNED`
- `FROZEN -> BANNED`

说明：

- `BANNED` 为终止态，一期不设计自动恢复

**用户激活触发机制：**

`PENDING_DEPOSIT → ACTIVE` 的完整跨服务链路如下：

1. `wallet-service` 链上确认完成，发布 `falconx.wallet.deposit.confirmed`
2. `trading-core-service` 幂等消费，执行入账，写 `t_account / t_ledger / t_deposit`
3. `trading-core-service` 完成入账后，通过 Outbox 发布 `falconx.trading.deposit.credited`（payload 含 `userId`）
4. `identity-service` 消费 `falconx.trading.deposit.credited`，若当前状态为 `PENDING_DEPOSIT`，迁移至 `ACTIVE`，更新 `t_user.activated_at`

约束：

- `identity-service` 消费时必须做幂等（用 `eventId` 去重）
- 若用户状态已为 `ACTIVE`，直接幂等跳过，不报错
- `FROZEN` 或 `BANNED` 用户收到此事件，不触发迁移（幂等返回原状态）

## 3. 钱包入金原始交易状态

owner：

- `wallet-service`

状态：

- `DETECTED`
- `CONFIRMING`
- `CONFIRMED`
- `REVERSED`
- `IGNORED`

允许迁移：

- `DETECTED -> CONFIRMING`
- `CONFIRMING -> CONFIRMED`
- `CONFIRMED -> REVERSED`
- `DETECTED -> IGNORED`
- `CONFIRMING -> IGNORED`

说明：

- `REVERSED` 用于链回滚或业务撤销
- `IGNORED` 用于无效或不归属平台的交易
- 未确认阶段若观察到链上回滚或交易消失，不直接进入 `REVERSED`，而是按业务语义处理为保留原状态或 `IGNORED`
- `confirmedAt` 的语义固定为“首次达到最终确认阈值的时间”，后续确认数增长时不得覆盖原值

## 4. 业务入金状态

owner：

- `trading-core-service`

状态：

- `CREDITED`
- `REVERSED`

允许迁移：

- `CREDITED -> REVERSED`

说明：

- 一期业务入账事实表只保留最终结果，不保留复杂中间态

## 5. 订单状态

owner：

- `trading-core-service`

状态：

- `PENDING`
- `TRIGGERED`
- `FILLED`
- `CANCELLED`
- `REJECTED`

允许迁移：

- `PENDING -> FILLED`
- `PENDING -> REJECTED`
- `PENDING -> CANCELLED`
- `PENDING -> TRIGGERED`
- `TRIGGERED -> FILLED`
- `TRIGGERED -> REJECTED`
- `TRIGGERED -> CANCELLED`

说明：

- 一期市价单主链路通常直接 `PENDING -> FILLED/REJECTED`
- `TRIGGERED` 为后续限价单和止损单预留
- 非交易时段只拒绝新的开仓请求；平仓和强平不受交易时间校验阻塞

## 6. 持仓状态

owner：

- `trading-core-service`

状态：

- `OPEN`
- `CLOSED`
- `LIQUIDATED`

允许迁移：

- `OPEN -> CLOSED`
- `OPEN -> LIQUIDATED`

说明：

- `CLOSED` 和 `LIQUIDATED` 都是终态
- `OPEN` 持仓允许发生“追加逐仓保证金”资金动作，但该动作不会触发状态迁移
- 手动平仓成功时写 `close_reason=1=manual`
- 手动平仓成功时同事务写本地 `t_outbox.event_type=trading.position.closed`
- `take_profit_price / stop_loss_price` 属于持仓级参数，允许在 `OPEN` 状态下修改
- 追加逐仓保证金成功时必须同步更新 `margin / liquidation_price`，并保持 `status=OPEN`
- 追加逐仓保证金当前不新增 Kafka 业务事件；审计事实以 `t_ledger.biz_type=10` 和持仓最新快照为准
- 非交易时段只拒绝新的开仓请求；已存在持仓的手动平仓不受交易时间校验阻塞
- 多头触发条件：
  - `effective_mark_price >= take_profit_price` 触发止盈
  - `effective_mark_price <= stop_loss_price` 触发止损
- 空头触发条件反向：
  - `effective_mark_price <= take_profit_price` 触发止盈
  - `effective_mark_price >= stop_loss_price` 触发止损
- `effective_mark_price` 的当前统一口径为：`BUY / 多头 -> bid`，`SELL / 空头 -> ask`
- 由 TP/SL 关闭的持仓，`close_reason` 分别记录为 `2=tp`、`3=sl`
- 手动平仓、强平执行与追加逐仓保证金必须统一使用 `SELECT FOR UPDATE` 锁定持仓 owner 记录
- 强平执行前必须基于最新 DB 持仓二次校验 `margin / liquidation_price`；若补仓后已不满足强平条件，则取消本次强平执行

### 6.1 净持仓模型冻结说明（SPEC-TRD-001）

- 单用户单 `symbol` 任一时刻最多只允许存在一个 `OPEN` 净持仓
- 同向下单：当前净仓加仓
- 反向下单且数量小于当前净仓：当前净仓减仓
- 反向下单且数量等于当前净仓：当前净仓减为 `0` 并进入终态
- 反向下单且数量大于当前净仓：先把当前净仓减为 `0`，剩余数量翻为反向净仓
- 净持仓模型下保留“每用户每 `symbol` 一个稳定 `positionId`”
- 后续正式实现中，加仓、减仓、穿零翻仓都在同一稳定 `positionId` 上演进，不额外创建新的独立持仓主键
- 本节为产品规则冻结，不代表当前仓库状态机实现已切换完成

## 7. 行情时效状态

owner：

- `market-service`

运行态：

- `FRESH`
- `STALE`

规则：

- 若当前时间减去 `quote.ts` 超过 `5s`，则视为 `STALE`
- `STALE` 必须按读取时的“当前时间 vs quote.ts”动态计算，不能只依赖缓存中写入时的布尔值

业务影响：

- 开仓默认拒绝使用 `STALE` 价格
- 当前已实现的手动平仓同样拒绝使用 `STALE` 价格，返回 `30002`
- 当前已实现的手动平仓在 Redis 无可用最新价时返回 `30003`
- 后续强平阶段继续沿用相同的价格来源审计要求

## 8. 状态变更实现要求

所有状态迁移代码都必须满足：

- 有清晰方法注释
- 有单元测试
- 非法迁移显式拒绝
- 关键迁移产生事件或审计记录
