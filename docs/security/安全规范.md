# FalconX v1 安全规范

## 1. 目标

统一 FalconX v1 的认证机制、授权规则、密钥管理和敏感数据处理要求。

本规范是实施阶段的强制规则。

## 2. JWT 规范

### 2.1 算法

使用 `RS256`（RSA + SHA-256 非对称签名）。

原因：

- `identity-service` 持有私钥，用于签名
- `gateway` 持有公钥，用于验证，无需向 `identity-service` 发起远程调用
- 比 `HS256` 更适合多服务场景，私钥不需要分发

禁止：

- 使用 `none` 算法
- 使用 `HS256`（共享密钥，任何持有密钥的服务都能伪造 token）

### 2.2 密钥管理

- RSA 密钥对长度：最小 `2048 bit`，推荐 `4096 bit`
- 私钥：只存在于 `falconx-identity-service` 的配置中，不得出现在其他服务或代码仓库
- 公钥：通过配置文件分发给 `falconx-gateway`
- 密钥存储：生产环境使用环境变量或密钥管理服务（如 Vault），不得硬编码

### 2.3 Token 结构

Access Token payload 至少包含：

- `sub`：用户主键 ID（`t_user.id`）
- `uid`：对外展示 UID（`t_user.uid`）
- `email`：脱敏邮箱（仅做展示参考，不用于业务判定）
- `status`：用户状态（`ACTIVE / FROZEN` 等）
- `iat`：签发时间（Unix timestamp）
- `exp`：过期时间（Unix timestamp）
- `jti`：JWT 唯一 ID，用于强制失效和审计

Refresh Token payload 至少包含：

- `sub`：用户主键 ID
- `jti`：Refresh Token 唯一 ID
- `iat`：签发时间
- `exp`：过期时间

### 2.4 Token 有效期

| Token 类型 | 有效期 | 说明 |
|---|---|---|
| Access Token | 15 分钟 | 短有效期，泄漏损失可控 |
| Refresh Token | 72 小时 | 用于换取新 Access Token，超出时必须重新登录 |

### 2.5 Token 刷新策略

- 客户端在 Access Token 过期前或过期后，使用 Refresh Token 调用刷新接口
- `identity-service` 验证 Refresh Token 有效性（未过期、未失效）
- 刷新成功后签发新的 Access Token 和新的 Refresh Token
- 一期 Refresh Token 采用轮换重签模式：旧 Token 立即失效，新 Token 重新获得完整 72 小时有效期
- Refresh Token 一次性使用：刷新后旧 Refresh Token 立即失效，签发新 Refresh Token

接口：

- `POST /api/v1/auth/refresh`，Body 传 `refreshToken`

### 2.6 Token 强制失效

需要立即失效的场景：

- 用户主动登出
- 管理员封禁用户（状态变为 `BANNED`）

实现方式：

- 将 `jti` 加入 Redis 黑名单，设置 TTL 与剩余有效期对齐
- gateway 验证 token 时，同时检查 Redis 黑名单

## 3. 认证流程

### 3.1 登录流程

```
Client --(POST /api/v1/auth/login)--> Gateway --(路由)--> identity-service
  identity-service: 验证邮箱 + bcrypt 密码
  identity-service: 检查用户状态（仅 ACTIVE 允许登录）
  identity-service: 签发 Access Token + Refresh Token
  identity-service: 返回 { accessToken, refreshToken, expiresIn }
```

登录状态规则：

- `PENDING_DEPOSIT`：返回 `10011 User Not Activated`
- `FROZEN`：返回 `10007 User Frozen`
- `BANNED`：返回 `10002 User Banned`
- 仅 `ACTIVE` 用户签发 Token

### 3.2 请求认证流程

```
Client --(Authorization: Bearer <accessToken>)--> Gateway
  Gateway: 用公钥验证 JWT 签名
  Gateway: 检查 exp 是否过期
  Gateway: 检查 jti 是否在 Redis 黑名单
  Gateway: 提取 sub/uid/status，注入请求 Header，路由到下游
  下游服务: 从 Header 读取 userId，不再重复验证 JWT
```

下游服务接收的 Header：

- `X-User-Id`：用户主键 ID
- `X-User-Uid`：用户 UID
- `X-User-Status`：用户状态
- `X-Trace-Id`：链路追踪 ID（由 gateway 内部生成并注入，不接受前端指定）

## 4. 授权规则

### 4.1 公开接口（无需 token）

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

### 4.2 认证接口（需要有效 Access Token）

其余所有 `/api/v1/**` 接口都需要认证。

gateway 拒绝条件：

- 无 `Authorization` Header
- token 格式非法
- token 签名验证失败
- token 已过期
- token `jti` 在黑名单中

拒绝时返回：

```json
{
  "code": "10001",
  "message": "Unauthorized",
  "data": null,
  "timestamp": "2026-04-16T16:00:00Z",
  "traceId": "..."
}
```

HTTP 状态码：`401`

### 4.3 状态限制

用户状态为 `FROZEN` 或 `BANNED` 时：

- `FROZEN`：可以查询，禁止写操作（下单、入金申请等）
- `BANNED`：拒绝所有请求，返回 `10002` 错误码

此逻辑在 gateway 层检查 `X-User-Status`，不需要下游服务重复判断。

## 5. 密码规范

- 算法：`bcrypt`，强度（cost factor）≥ `12`
- 密码长度要求：`8-64` 字符
- 注册时只存储哈希值，明文密码不得持久化
- 禁止在日志中打印明文密码、密码哈希

## 6. 速率限制

### 6.1 登录失败限制

- 同一 IP，连续登录失败 `5` 次：锁定 `15` 分钟
- 锁定期间返回 `10003` 错误码，不透露具体原因

实现方式：

- Redis 计数器，key 为 `falconx:auth:login:fail:{ip}`，TTL `15m`

### 6.2 注册限流

- 同一 IP，`1` 小时内最多注册 `5` 次
- 超过限制返回 `10004` 错误码

### 6.3 全局限流

- 一期暂不强制，后续根据运营情况在 gateway 层补充

## 7. 敏感数据处理规范

### 7.1 禁止明文存储

- 密码：必须 bcrypt 哈希后存储
- 私钥、API Key：使用密钥管理服务，不得在代码或配置文件中硬编码

### 7.2 禁止日志打印

详见[日志打印规范](../architecture/日志打印规范.md) §6。

额外强调：

- JWT 全量内容（含 payload）禁止完整打印
- 只允许打印 `jti` 或 token 前缀（如前 8 位）
- Refresh Token 禁止打印

### 7.3 传输安全

- 生产环境所有 HTTP 通信必须走 `HTTPS / TLS`
- 一期开发环境可用 HTTP，但需明确标注不得用于生产

## 8. CORS 规范

- 一期前端域名固定配置白名单
- 不使用 `Access-Control-Allow-Origin: *`
- 允许的请求头：`Authorization, Content-Type, X-Request-Id, X-Idempotency-Key`
- 允许的方法：`GET, POST, PUT, PATCH, OPTIONS`

## 9. 测试要求

安全相关代码必须覆盖：

- 正常登录和 token 验证路径
- `PENDING_DEPOSIT` 用户登录被拒绝
- 过期 token 被拒绝
- 黑名单 token 被拒绝
- 密码错误触发计数
- 超过失败次数后锁定
- `BANNED` 用户被拒绝
- Refresh Token 一次性使用（第二次使用旧 token 被拒绝）
