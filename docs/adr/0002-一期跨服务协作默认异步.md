# ADR-0002 一期跨服务协作默认异步

## 状态

已采纳

## 背景

微服务之间最容易失控的点，是到处出现同步 HTTP 互调。

在 FalconX v1 中，若 `identity / market / trading-core / wallet` 形成网状互调，会导致：

- 服务边界模糊
- 请求链变长
- 故障传播面扩大
- 重试与幂等复杂度上升

## 决策

一期默认不建设常规服务间同步互调。

跨服务业务协作优先采用：

- `Kafka`
- `Redis`

北向流量仍遵守：

- `Client -> Gateway -> Owning Service`

## 结果

一期主要协作方式固定为：

- `market-service -> Redis + Kafka`
- `wallet-service -> Kafka`
- `trading-core-service` 消费市场和钱包事件
- `gateway` 只承担入口和聚合查询

## 例外

如果后续确实出现必须同步调用的场景，可以采用：

- `RestClient`
- `HTTP Service Interface Clients`

但必须满足：

- 有明确 owner
- 没有更合适的事件化方式
- 在文档中写明调用原因和失败策略

## 后果

优点：

- 服务耦合度更低
- 系统更容易扩展
- 交易核心链路不依赖跨服务同步响应

代价：

- 某些跨域流程会变成最终一致
- 需要更清晰的事件与幂等设计
