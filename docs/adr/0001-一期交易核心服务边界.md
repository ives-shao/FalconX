# ADR-0001 一期交易核心服务边界

## 状态

已采纳

## 背景

FalconX 是 CFD `B-book` 模式，交易主链路包含：

- 下单
- 保证金占用
- 手续费扣减
- 持仓变更
- 成交落库
- 强平执行

如果一期把 `trade / fund / risk` 拆成多个独立服务，会立刻引入：

- 分布式事务问题
- Saga 编排复杂度
- 订单处理中间态
- 更多跨服务同步调用

这与一期“先求稳定闭环”的目标冲突。

## 决策

一期把下面三类能力合并到一个服务：

- `trade`
- `fund`
- `risk`

统一形成：

- `falconx-trading-core-service`

## 结果

`trading-core-service` 成为一期的同步交易事务边界，拥有：

- 账户
- 账本
- 业务入金事实
- 订单
- 持仓
- 成交
- 风控参数
- 强平日志

对应 owner schema：

- `falconx_trading`

## 后果

优点：

- 下单和平仓仍可用本地事务完成
- 账务与持仓强一致
- 一期实现复杂度明显下降

代价：

- 服务粒度比完全拆分更粗
- 二期如果流量或团队规模上来，需要再拆出更细服务

## 后续约束

- 一期不再单独建设 `fund-service`、`trade-service`、`risk-service`
- 如果后续要拆分，必须先新增 ADR 说明拆分原因、边界和迁移路径
