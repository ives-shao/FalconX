# FalconX v1 日志打印规范

## 1. 目标

统一 FalconX v1 各服务的日志输出规则，确保后续出现问题时可以快速定位：

- 请求从哪里进入
- 业务执行到哪一步
- 哪个状态发生了变化
- 哪个外部依赖失败
- 哪个业务键需要被追踪

本规范是实施阶段的强制规则。

## 2. 总体原则

- 关键业务节点必须打日志
- 日志必须可关联请求和业务实体
- 日志必须避免泄露敏感数据
- 日志要为排障服务，不写无意义噪音
- 同一类事件使用统一字段和统一话术

## 3. 必带关联字段

所有关键日志都应尽量带上下面的关联字段：

- `traceId`
- `requestId`
- `service`
- `module`
- `event`

涉及业务实体时，按场景补充：

- `userId`
- `uid`
- `symbol`
- `orderNo`
- `positionId`
- `accountId`
- `txHash`
- `chain`
- `eventId`
- `topic`

规则：

- 没有业务实体时，至少保留 `traceId`
- 涉及交易或资金时，必须至少带一个业务主键

补充规则：

- `traceId` 必须通过 `MDC` 或等价日志上下文机制自动输出
- 不允许在每条日志 message 中手工拼接 `traceId={}` 这类重复内容
- 日志模板应统一自动带出 `traceId`

## 3.1 TraceId 生成与传递规则

固定规则：

- `traceId` 只由服务端生成
- 外部入口统一由 `gateway` 生成
- 内部服务收到请求后继续透传，不重新生成
- Kafka 事件由生产者沿用当前链路 `traceId`
- 定时任务、补偿任务、重放任务在没有上游链路时自行生成新的 `traceId`

禁止事项：

- 不支持前端传入 `traceId`
- 即使客户端传入 `X-Trace-Id`，一期默认也忽略
- 不允许多个独立请求复用同一个前端指定 `traceId`

推荐实现：

- 入口过滤器生成 `traceId`
- 将 `traceId` 放入 `MDC`
- 日志框架 pattern 自动打印 `traceId`
- HTTP client、Kafka producer 自动透传当前 `traceId`

## 4. 日志级别规范

### `INFO`

用于记录关键业务事实：

- 请求进入和结束
- 状态迁移成功
- 外部系统连接成功
- 重要业务动作完成

### `WARN`

用于记录可恢复但异常的情况：

- 参数非法
- 重复请求
- 行情 stale
- 幂等冲突
- 外部依赖超时后重试
- 非法状态迁移尝试

### `ERROR`

用于记录需要排障的失败：

- 数据库事务失败
- Kafka 发布失败
- 消费失败且无法恢复
- 强平执行失败
- 链上监听异常
- 网关路由或鉴权异常

### `DEBUG`

仅用于开发调试：

- 低价值细节
- 中间计算过程
- 第三方响应裁剪版

规则：

- 生产环境默认不开启高频 `DEBUG`

## 5. 日志内容规范

每条关键日志至少应包含：

- 事件名称
- 执行结果
- 核心业务键
- 必要上下文

推荐格式：

```text
event=order_created result=success traceId=... userId=... orderNo=... symbol=BTCUSDT
```

不要只写：

```text
create order success
```

原因：

- 这种日志无法定位具体实体

补充说明：

- 示例中出现 `traceId=...` 只是说明日志最终呈现效果
- 实际编码时不应在业务代码里手工拼接 `traceId`
- 正确做法是通过日志模板自动输出

## 6. 敏感信息规范

下面内容禁止原样打印：

- 明文密码
- JWT 全量内容
- 私钥
- API Key
- 完整邮箱
- 完整手机号
- 完整钱包私密数据

允许的处理方式：

- 邮箱脱敏：`iv***@example.com`
- 地址脱敏：显示前 6 位和后 4 位
- Token 只打印前缀或 jti

## 7. 关键业务节点日志要求

## 7.1 `gateway`

必须打印：

- 请求进入
- 鉴权成功或失败
- 路由目标
- 下游调用耗时
- 聚合查询完成
- 请求异常结束

建议字段：

- `traceId`
- `requestId`
- `path`
- `method`
- `targetService`
- `status`
- `latencyMs`

## 7.2 `identity-service`

必须打印：

- 注册成功
- 注册失败
- 登录成功
- 登录失败
- token 刷新
- 用户状态变更

建议字段：

- `traceId`
- `userId`
- `uid`
- `maskedEmail`
- `event`
- `statusBefore`
- `statusAfter`

## 7.3 `market-service`

必须打印：

- 外部行情连接成功
- 连接断开和重连
- 报价标准化失败
- stale 判定
- K 线落盘
- 价格事件发布失败

建议字段：

- `traceId`
- `symbol`
- `source`
- `quoteTs`
- `stale`
- `topic`

## 7.4 `trading-core-service`

必须打印：

- 下单请求进入
- 风控校验结果
- 保证金冻结/确认
- 手续费扣减
- 订单创建
- 订单拒绝
- 订单成交
- 持仓创建或变更
- 平仓完成
- 强平触发
- 强平执行结果
- 入金入账成功或失败

建议字段：

- `traceId`
- `userId`
- `symbol`
- `orderNo`
- `positionId`
- `accountId`
- `margin`
- `fee`
- `pnl`
- `price`
- `priceSource`

## 7.5 `wallet-service`

必须打印：

- 地址分配
- 新交易检测
- 确认数推进
- 交易确认完成
- 回滚或忽略
- 事件发布失败

建议字段：

- `traceId`
- `userId`
- `chain`
- `txHash`
- `address`
- `confirmations`
- `requiredConfirms`

## 8. 外部调用日志规范

调用外部系统或关键中间件时，至少打印：

- 调用目标
- 调用类型
- 结果
- 耗时
- 重试次数

适用场景：

- Tiingo
- 链节点 RPC
- Kafka 发布
- Redis 关键失败

## 9. 状态迁移日志规范

任何状态迁移都必须打印：

- `event`
- `entity`
- `id`
- `fromStatus`
- `toStatus`
- `result`

适用对象：

- 用户
- 链上交易
- 业务入金
- 订单
- 持仓

## 10. 异常日志规范

异常日志必须说明：

- 发生在哪个步骤
- 影响哪个业务实体
- 是否会重试
- 下一步动作是什么

推荐格式：

```text
event=wallet_deposit_confirm_failed result=error traceId=... chain=ETH txHash=... retry=true reason=...
```

## 11. 实现要求

后续代码实现时必须满足：

- 入口层统一注入 `traceId` 和 `requestId`
- 服务层关键节点打 `INFO`
- 风险和异常分支打 `WARN/ERROR`
- 所有日志使用统一键风格
- 不允许在关键链路完全无日志
- `traceId` 通过 `MDC` 自动打印，不在业务日志 message 中手工拼接

## 12. 测试要求

日志虽然不是主业务结果，但关键链路测试应至少验证：

- 关键节点有日志输出
- 日志不包含敏感信息
- traceId 能跨链路透传
