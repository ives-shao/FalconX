# FalconX v1 事务与幂等规范

## 1. 目标

统一 FalconX v1 的本地事务边界、最终一致策略和幂等控制点。

## 2. 基本原则

- 一期不引入分布式事务
- 服务内一致性使用本地事务
- 跨服务一致性使用事件和幂等保证
- 所有关键写操作都必须可重试

## 3. 本地事务边界

### `identity-service`

本地事务覆盖：

- 注册用户
- 修改用户状态（含 `PENDING_DEPOSIT → ACTIVE` 激活迁移）
- 记录登录时间

### `market-service`

本地事务覆盖：

- 市场元数据维护
- 交易时间与节假日规则维护
- 低频市场事件的 `t_outbox` 写入

说明：

- Redis 最新价更新不进 MySQL 事务
- 交易时间规则快照写 Redis 不进 MySQL 事务，但必须由 owner 数据变更触发刷新
- 报价历史与 K 线历史写入 ClickHouse
- `falconx.market.price.tick` 直接发 Kafka，不经过 `t_outbox`
- `falconx.market.kline.update` 和低频市场元数据事件通过 `t_outbox` 与本地事务衔接

### `trading-core-service`

本地事务覆盖：

- 账户校验
- 保证金冻结或确认
- 手续费扣减
- 订单写入
- 持仓写入
- 净敞口写入
- 成交写入
- 账本写入
- 强平日志写入

说明：

- `trading-core-service` 是一期最重要的事务边界
- `QuoteDrivenEngine` 的高频读路径只允许读取内存持仓快照和 Redis 最新价，不允许按 tick 直查 MySQL

### `wallet-service`

本地事务覆盖：

- 钱包地址分配
- 原始链交易写入
- 确认数推进
- 游标更新

## 4. 跨服务一致性

跨服务统一采用：

- `本地事务 + Outbox`
- Kafka 事件投递
- 消费端幂等

一期 Outbox 固定采用：

- 服务内定时轮询 `t_outbox`
- 不是 Debezium CDC
- 不是应用代码里直接同步发 Kafka 后再补写数据库
- 公共组件统一放在 `falconx-infrastructure` 的 `outbox` / `inbox` 包中

### 4.1 Outbox 轮询机制

实现路线固定为：

1. 业务事务内写入 `t_outbox`
2. 每个服务本地启动 `OutboxPublisherScheduler`
3. 调度器按固定频率批量轮询待发送事件
4. 成功后标记 `sent`
5. 失败后更新 `retry_count / next_retry_at / last_error`
6. 达到最大重试次数后标记 `dead`

一期推荐参数：

- 轮询间隔：`1s`
- 单批条数：`200`
- 最大重试次数：`10`
- 退避基线：`5s -> 30s -> 120s -> 30m`
- 每档统一叠加 `±20%` jitter，避免同批失败消息在固定时间同时重试
- 超过阈值：标记 `dead`，由运维或补偿任务介入处理

推荐公共包结构：

- `com.falconx.infrastructure.outbox`：`OutboxMessage`、`OutboxRepositorySupport`、`OutboxPublisherScheduler`
- `com.falconx.infrastructure.inbox`：`InboxMessage`、`InboxRepositorySupport`、`InboxIdempotencyGuard`

推荐调度流程：

- 使用 `SELECT ... FOR UPDATE SKIP LOCKED` 选取一批 `pending/failed`
- 在短事务内把状态更新为 `dispatching`
- 提交事务后再调用 Kafka producer
- 成功回写 `sent`
- 失败回写 `failed`

消费者幂等固定采用：

- 低频关键事件使用消费端 `t_inbox` 去重表
- 高频价格 tick 不写 `t_inbox`，幂等由处理逻辑本身或 Redis 短窗口保证
- `market.kline.update` 属于低频关键事件，`trading-core-service` 消费后必须通过 owner `Mapper + XML + Repository` 写入 `t_inbox`
- `eventId` 唯一约束只强制用于低频关键事件
- `t_inbox` 只能在“业务处理成功”或“幂等跳过成立”之后写入；若 owner 业务对象不存在或业务动作未完成，必须让消费失败并走重试，不能提前标记成功

禁止：

- 依赖跨服务同步调用完成一整个业务事务
- 依赖人工补偿作为默认路径

## 5. 幂等键分层

### 5.1 HTTP 写请求

使用：

- `X-Idempotency-Key`

适用：

- 注册
- 下单
- 平仓
- 地址申请

### 5.2 交易业务

使用：

- `clientOrderId`

适用：

- 下单请求去重

### 5.3 钱包链上交易

使用：

- `walletTxId`

适用：

- wallet owner 对外输出的稳定原始交易标识
- trading-core 业务入账事实去重

补充约束：

- `wallet-service` 内部的原始链事实可按 `(chain, tx_hash, log_index)` 拆分
- 跨服务 confirmed / reversed 事件不得再直接以 `(chain, tx_hash)` 作为最终业务幂等键
- 原生币转账固定使用 `log_index = 0`

### 5.4 账本流水

使用：

- `t_ledger.idempotency_key`

适用：

- 资金动作幂等

### 5.5 Kafka 消费

使用：

- `eventId`

适用：

- 事件消费去重

## 6. 一期关键流程要求

### 6.1 下单

规则：

- 接口层校验 `X-Idempotency-Key` 或 `clientOrderId`
- 交易时间校验必须先于保证金冻结执行
- 应用层进入本地事务
- 事务内完成账户、订单、持仓、成交、账本和 `t_risk_exposure` 写入
- 持仓若携带 `takeProfitPrice / stopLossPrice`，必须和开仓一起持久化到 `t_position`
- 事务后通过 `Outbox` 发布订单事件

### 6.2 入金确认

规则：

- `wallet-service` 先确认原始链交易状态
- 生成 `deposit.confirmed` 事件
- `trading-core-service` 幂等消费
- 事务内完成账户余额与账本写入
- 同一事务内写入 `falconx.trading.deposit.credited` 对应的 `t_outbox`

### 6.3 用户激活

规则：

- `trading-core-service` 完成入账后，事务内写 `t_outbox`，投递 `falconx.trading.deposit.credited`
- `identity-service` 幂等消费（用 `eventId` 去重）
- 事务内检查用户当前状态，若为 `PENDING_DEPOSIT`，迁移至 `ACTIVE`，更新 `t_user.activated_at`
- 若用户尚未落库或 owner 业务对象不存在，必须抛错并等待事件重试，不允许把该场景记为已消费
- 幂等键：`eventId`（消费端 Redis 或 DB 去重均可，使用 DB 唯一约束更稳）

### 6.4 实时触发与强平

规则：

- `trading-core-service` 消费价格事件
- 高频 `price.tick` 不落 `t_inbox`
- `price.tick` 的 Kafka 入口允许容器级失败重试，但不得把高频链路改造成 `t_inbox` 持久化消费
- 服务启动时从 DB 装载全部 `OPEN` 持仓到内存快照，并按 `symbol` 分组索引
- 开仓、平仓、强平、TP/SL 修改后必须同步更新内存快照
- `QuoteDrivenEngine` 运行时只读内存快照，不直接查 MySQL
- 对订单或持仓做业务级幂等判定
- 如需短窗口去重，优先使用 Redis，避免为 tick 写数据库去重表
- 防止同一价格事件重复触发相同业务动作

### 6.5 Swap 结算

规则：

- `market-service` 维护 `t_swap_rate`
- `trading-core-service` 按 `rollover_time` 定时扫描 `OPEN` 持仓
- 结算时必须使用 rollover 时点的 fresh 有效价计算结算名义本金
- `Swap` 结算金额 = 结算名义本金 × 多/空费率
- 结算金额写 `t_ledger`，必要时写 `t_outbox`
- 对应账本业务类型固定使用：
  - `6=swap_charge`
  - `7=swap_income`
- 一期不引入三倍日和节假日顺延特殊规则
- 本节是正式规则冻结，不代表当前仓库已实现 `Swap`

### 6.6 负净值保护

规则：

- 强平导致实际亏损超过账户可承受范围时，账户余额最低只能到 `0`
- 超出的平台兜底金额记录到 `t_liquidation_log.platform_covered_loss`
- 不允许出现负余额账户
- 穿仓不形成用户负债，不追收
- 本节已完成规则冻结；当前仓库只在已有实现范围内体现该口径

### 6.7 交易时间校验

规则：

- `market-service` owner 持久化 `t_trading_hours / t_trading_hours_exception / t_trading_holiday`
- `market-service` 启动时把每个 `symbol` 的交易时间快照写入 Redis
- Redis 交易时间快照固定设置 `25h` TTL，并由 `market-service` 每日 `UTC 00:00` 定时全量刷新
- `trading-core-service` 下单前只读 Redis 快照，不跨服务读取 `falconx_market`
- Redis 快照 miss 时默认拒绝开仓，避免在规则不明时放行订单
- 判定优先级固定为：
  1. `t_trading_hours_exception`
  2. `t_trading_holiday`
  3. `t_trading_hours`
- 跨午夜 session 允许使用单条规则表达，运行时必须正确支持“开盘日在前一日、收盘在次日凌晨”的窗口判定
- 非交易时段返回 `40008: Symbol Trading Suspended`

## 7. 非法实现示例

下面做法明确禁止：

- 网关直接写业务库
- 一个 HTTP 请求横跨多个服务数据库做事务
- 没有幂等键的下单写接口
- 消费 Kafka 时不做 `eventId` 去重
- 链上交易只靠 Redis 去重、不做数据库唯一约束
- 业务代码直接调用 Kafka producer 代替 Outbox

## 8. 测试要求

事务与幂等相关代码必须至少覆盖：

- 正常路径
- 重复请求路径
- 重复消息路径
- 事务中途失败路径
- 重试后成功路径
