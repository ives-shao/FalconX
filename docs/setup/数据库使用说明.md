# 数据库使用说明

## 1. 目的

统一 FalconX 本地开发、集成测试与临时调试时的数据库命名和使用口径，避免出现“看到很多库但不知道哪个才是标准库”的情况。

## 2. 标准口径

FalconX 当前只承认两类标准库：

### 2.1 运行库

用于本地正常启动服务、联调接口和观察真实运行链路。

| 服务 | 标准 MySQL Schema |
|---|---|
| `identity-service` | `falconx_identity` |
| `market-service` | `falconx_market` |
| `trading-core-service` | `falconx_trading` |
| `wallet-service` | `falconx_wallet` |

ClickHouse 标准分析库固定为：

- `falconx_market_analytics`

### 2.2 测试库

用于 `SpringBootTest`、集成测试和持久化测试。

| 服务 | 标准测试库 |
|---|---|
| `identity-service` | `falconx_identity_it` |
| `market-service` | `falconx_market_it` |
| `trading-core-service` | `falconx_trading_it` |
| `wallet-service` | `falconx_wallet_it` |

约束：

- 标准测试库统一使用 `_it` 后缀
- 测试库不作为人工联调或长期运行服务的默认库
- `market-service` 测试启动前会自动重建 `falconx_market_it`，随后走当前 migration 的标准迁移路径，确保不会被历史 checksum 残留污染

## 3. 非标准库说明

以下库名不属于仓库标准基线：

- `*_debug`
- 历史临时库
- 人工排障时自行指定的临时库

说明：

- 这类库通常来自手工执行 `java -jar ... --spring.datasource.url=...` 的命令行覆盖
- 仓库不会把它们当作默认运行库
- 若本机存在这类库，只能视为“个人临时调试痕迹”，不应作为团队共享口径

## 4. 如何判断当前到底用了哪个库

按下面优先级判断：

1. 启动命令显式传入的 `--spring.datasource.url`
2. 测试类上的 `@SpringBootTest(properties = ...)`
3. 服务自身 `src/main/resources/application.yml`

也就是说：

- 如果你直接启动服务且没有额外覆盖，默认使用运行库
- 如果你运行测试，默认使用对应 `_it` 测试库
- 如果你手工传了 `--spring.datasource.url=..._debug`，则实际生效的是 `_debug`

## 5. 当前仓库结论

当前仓库的标准数据库使用应理解为：

- 本地运行：`falconx_identity / falconx_market / falconx_trading / falconx_wallet`
- 测试执行：`falconx_identity_it / falconx_market_it / falconx_trading_it / falconx_wallet_it`
- `*_debug`：仅本机手工调试使用，不属于标准配置

## 6. 使用建议

后续若需要手工联调：

- 默认只连运行库
- 运行测试只看 `_it` 测试库
- 若使用 `*_debug`，必须明确知道是自己手工覆盖出来的临时库
- 清理历史库时，优先保留运行库与 `_it` 测试库，其他临时库可按需删除
