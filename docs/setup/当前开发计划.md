# FalconX 当前开发计划

## 1. 当前项目判断

- FalconX `v1` 采用 `GODSA`，一期固定服务为 `gateway / identity-service / market-service / trading-core-service / wallet-service`。
- `Stage 1-5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、ClickHouse、Kafka 与最小北向链路已切到真实实现。
- `Stage 6A` 主链路收口已完成，`Stage 6B` 当前冻结范围已完成并收口。
- 当前正式阶段结论：`Stage 6C Jackson 3 专项迁移` 已完成并收口。
- `Stage 7 / 7A` 仍冻结，待你拍板是否解冻。
- 当前系统仍不能表述为“生产可用”或“可安全对外公测”。

## 9. 当前风险与冻结项

- `README / 开发启动手册 / 当前开发计划 / 测试规范 / 事务与幂等规范` 若不同步，会造成 `Swap` owner 共享口径和阶段口径漂移。
- Tiingo 与 wallet 外部自动化入口仍依赖显式环境变量与 JVM trust store；缺失时不会形成可归档真跑证据。
- `Swap` 当前采用 owner Redis 快照；若未来扩展 JSON 结构，必须继续保证交易侧反序列化兼容。
- `Stage 7` 的 `TC-LOG-002` 正式定义已收敛为“同一业务链路/同一事件链路跨服务 `traceId` 一致”；当前不再受“单一北向请求同时贯穿 gateway / market / trading”定义冲突阻塞，但 `Stage 7` 仍未完成统一回归与压测归档。
- `Swap` 当前已发布 `falconx.trading.swap.settled`；若后续扩展 payload，必须保持加性兼容并补生产者/消费者联通验证。
- 当前只有 `ws://{host}/ws/v1/market` 被正式冻结；若后续扩展用户侧实时推送，必须先冻结新的 WebSocket 契约。
- `Swap` 当前不做精确历史补算；超出 `rollover ± stale.max-age` 的后续报价不得拿来补算同一 `rollover`。

### SPEC-TRD-001 规则冻结结论

- 持仓模式正式冻结为 `One-Way + Isolated`。
- 单用户单 `symbol` 只允许存在一个净持仓。
- 手续费冻结为双边收费；一期先以 `t_symbol.taker_fee_rate` 作为统一动态费率。
- `Swap` 冻结为按 `rollover` 时点 fresh 有效价计算结算名义本金并乘以多/空费率；一期不引入三倍日和节假日顺延特殊规则。
- `Swap rate` owner 共享链当前实现为 `market-service -> Redis snapshot -> trading-core-service`。
- 上述内容是正式产品规则与当前实现口径；`Stage 6B` 当前冻结范围已完成，但这不等于后续用户侧实时推送端点已经冻结或系统达到生产可用。
