# FalconX 当前开发计划

## 1. 当前项目判断

- FalconX `v1` 采用 `GODSA`，一期固定服务为 `gateway / identity-service / market-service / trading-core-service / wallet-service`。
- `Stage 1-5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、ClickHouse、Kafka 与最小北向链路已切到真实实现。
- 当前正式执行阶段固定为：`Stage 6A 收口专项`。
- `Stage 6B / 6C / 7 / 7A` 当前均未进入验收完成状态；即使 `main` 上存在部分超前代码或接口事实，也不改变当前正式阶段仍为 `Stage 6A` 的口径。
- 当前系统仍不能表述为“生产可用”或“可安全对外公测”。

## 9. 当前风险与后续冻结项

- `README / 开发启动手册 / 当前开发计划 / 统一接口文档` 若不同步，会造成阶段口径与验收口径漂移
- `wallet-service` 当前保留多链装配事实，不能把“仅配置限制 ETH 验证”误写成“已单链化”
- `market.price.tick` 属于高频 Kafka 链路，补失败重试专项时最容易影响 TP / 强平 / 浮盈亏链路
- `market.kline.update` 本轮新增 `trading` 侧消费，会带来幂等、重试、日志和文档联动风险
- 在存在本地 TLS inspection 的网络环境中，Tiingo 真连接仍依赖为 JDK 配置对应根 CA trust store；缺少该配置时会回到 `PKIX path building failed`
- Tiingo 真源自动化测试已改为显式环境门禁；若未提供 `FALCONX_MARKET_TIINGO_EXTERNAL_TEST_ENABLED=true`、`FALCONX_MARKET_TIINGO_API_KEY` 与可选 trust store，该组用例只会跳过，不会形成外部验证证据
- wallet 真节点自动化测试已改为显式环境门禁；若未提供 `FALCONX_WALLET_EXTERNAL_TEST_ENABLED=true`、`FALCONX_WALLET_ETH_RPC_URL` 与可选 JVM trust store，该组用例只会跳过，或在 TLS inspection 环境里回到 `PKIX path building failed`
- `Swap / 隔夜利息` 的产品规则已在 `SPEC-TRD-001` 冻结，但实现、测试与阶段验收仍未开始，当前不得误写为“已实现”

### SPEC-TRD-001 规则冻结结论

- 本轮只完成产品规则冻结，不涉及代码实现
- 持仓模式正式冻结为 `One-Way + Isolated`
- 单用户单 `symbol` 只允许存在一个净持仓
- 不允许多张同方向独立逐仓仓位
- 净持仓模型下保留“每用户每 `symbol` 一个稳定 `positionId`”
- `t_position` 冻结为净持仓主事实表；任一时刻同一用户同一 `symbol` 最多只有一条 `OPEN` 持仓
- 同向下单视为加仓，反向下单视为减仓；若反向数量超过当前净仓，则先减到 `0`，剩余部分翻为反向净仓
- 有效最大杠杆冻结为 `min(t_symbol.max_leverage, t_risk_config.max_leverage)`
- 手续费冻结为双边收费；一期先以 `t_symbol.taker_fee_rate` 作为统一动态费率
- 账户结算币一期固定 `USDT`；在正式折算链路落地前，只允许 `quote_currency ∈ {USD, USDT, USDC}` 的品种进入交易放行
- `Swap` 冻结为按 rollover 时点 fresh 有效价计算结算名义本金并乘以多/空费率；一期不引入三倍日和节假日顺延特殊规则
- 负余额保护冻结为账户余额下限 `0`
- 穿仓冻结为平台兜底，不形成用户负债、不追收
- 产品配置共享链冻结为 `market-service -> Kafka -> trading-core-service`
- 上述内容是正式产品规则冻结，不等于当前仓库已经实现
