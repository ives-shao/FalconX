# FalconX 当前开发计划

## 1. 当前项目判断

- FalconX `v1` 采用 `GODSA`，一期固定服务为 `gateway / identity-service / market-service / trading-core-service / wallet-service`。
- `Stage 1-5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、ClickHouse、Kafka 与最小北向链路已切到真实实现。
- 当前正式执行阶段固定为：`Stage 6A 收口专项`。
- `Stage 6B / 6C / 7 / 7A` 当前均未进入验收完成状态；即使 `main` 上存在部分超前代码或接口事实，也不改变当前正式阶段仍为 `Stage 6A` 的口径。
- 当前系统仍不能表述为“生产可用”或“可安全对外公测”。

## 2. 基线与分支结论

### 2.1 分支差异结论

- `origin/main` 是本轮唯一真源。
- `main` 当前基线为 `a5faa35`，与 `origin/main` 一致。
- `dev / origin/dev` 仅落后一个 merge commit，功能树 diff 为空；本轮不从旧分支回收功能代码。
- 本轮实施分支基于 `origin/main` 创建，禁止把旧分支上的过时代码口径带入当前阶段结论。

### 2.2 主文档冲突矩阵

| 主题 | 冲突前现状 | 本轮统一口径 |
|---|---|---|
| 当前正式阶段 | `README / 开发启动手册` 偏向 `Stage 6A`，`当前开发计划` 曾写成 `Stage 6B` | 当前正式阶段固定为 `Stage 6A 收口专项` |
| `Stage 6B` 定义 | 存在“安全完整化”和“用户侧实时与运营完整性”两套口径 | `Stage 6B` 统一保留为后续阶段，当前不启动、不验收 |
| `Stage 7 / 7A` 状态 | 旧文档与超前实现事实混用，易误判为已验收 | 本轮不把任何 `Stage 7 / 7A` 事实写成当前阶段完成结论 |
| 逐仓状态 | 文档与代码事实存在错位 | 逐仓相关超前实现不计入本轮阶段完成，`Stage 7A` 保持冻结 |
| 代表性 E2E 边界 | `统一接口文档` 曾只写 `gateway + identity + trading + Kafka` | 统一写为 `gateway + identity + market + trading + wallet + Kafka` 的受控真运行时联调 |
| `wallet` 单入口边界 | 文档要求单入口，代码保留多链装配 | 统一口径为：本轮仅配置限制 `ETH` 验证，不表述为代码已单链化 |
| `Swap / 隔夜利息` | 文档中既被放进 `Stage 6A`，又被带进后续阶段 | 本轮统一为 `Stage 6A` 待确认项，不进入实现 |

## 3. 阶段完成概览

- `Stage 1` 已完成：共享模块、多模块父工程、5 个服务与 contract 基础结构已落地。
- `Stage 2A` 已完成：`market-service` 市场数据底座已落地。
- `Stage 2B` 已完成：`wallet-service` 钱包原始事实底座已落地。
- `Stage 3A` 已完成：`identity-service` 身份域基础闭环已落地。
- `Stage 3B` 已完成：`trading-core-service` 交易核心基础闭环已落地。
- `Stage 4` 已完成：`gateway` 与各服务的最小北向链路已打通。
- `Stage 5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、Redisson、ClickHouse、Kafka 已切到真实实现。
- `Stage 6A` 部分完成：当前是唯一正式执行阶段。
- `Stage 6B / 6C / 7 / 7A` 当前全部冻结，待 `Stage 6A` 收口后再重新确认顺序与边界。

## 4. 当前仓库事实摘要

说明：

- 下列内容是 `origin/main` 上经代码与文档抽查确认的仓库事实，不等于对应阶段已经验收完成。
- 当前阶段完成判定只看 `Stage 6A` 范围，不把其他阶段的超前实现事实直接表述为“阶段已完成”。

### 已落地的仓库事实

- `identity-service`
  - 已提供 `register / login / refresh / logout`
  - 已消费 `falconx.trading.deposit.credited` 推进用户激活
  - 已具备登录失败锁定、Access Token 黑名单与 `logout -> blacklist -> gateway reject` 基础能力
- `gateway`
  - 已建立 JWT 鉴权、`traceId` 透传、统一 fallback、限流、熔断与超时配置
- `market-service`
  - 已接入 Tiingo JDK WebSocket 真连接代码路径
  - 已支持 owner `t_symbol.status=1` 白名单热刷新
  - 已落地最新价缓存、`quote_tick`、真实 K 线收盘聚合、ClickHouse 写入与 `market.kline.update` Outbox 投递
- `wallet-service`
  - 已落地钱包地址分配、owner 持久化、EVM 原生币与 ERC20 最小扫块识别路径
  - 已支持确认窗口重扫、reversal 观察与 `walletTxId` 稳定主键输出
- `trading-core-service`
  - 已落地 `wallet.deposit.confirmed / reversed` 与 `market.price.tick` 的正式消费入口
  - 已具备 `walletTxId` 幂等、Outbox / Inbox 基础链路
  - 已存在部分手动平仓、TP/SL、强平、逐仓相关实现事实，但这些事实不构成本轮阶段验收结论

## 6. Stage 6A 本轮冻结边界

- 当前阶段名称：`Stage 6A 收口专项`
- 当前阶段目标：在不进入 `Stage 6B / 6C / 7 / 7A` 的前提下，完成 `Stage 6A` 主链路收口
- 本轮固定口径：
  - `wallet-service` 仅按配置限制 `ETH` 验证，不做代码层单链收缩
  - `market.price.tick` 本轮补证范围固定为 `Kafka 入口失败重试专项`
  - `market.kline.update` 本轮要求在 `trading-core-service` 侧必须消费
  - 本轮阶段结论允许写成：`除 Swap 外主链路收口完成，Swap 保持待确认`
- 本轮不做：
  - `Stage 6B / 6C / 7 / 7A` 的功能开发与阶段验收
  - `Swap / 隔夜利息` 的默认实现
  - 新链类型、新 schema、新配置模型扩展

## 8. 验收口径

- 本轮完成标准仍为：`代码实现完成 + 测试结论明确 + 文档同步完成`
- `Stage 6A` 的最终结论只允许写成：
  - `除 Swap 外主链路收口完成，Swap 保持待确认`
- 必须同时具备：
  - `mvn clean compile` 通过
  - 目标测试通过
  - `docker compose config` 通过
  - 本地依赖可启动
  - Tiingo 真连接证据
  - 至少一条 ETH 真 RPC 证据
  - `market.price.tick` Kafka 入口失败重试专项证据
  - `market.kline.update` 在 `trading-core-service` 侧消费证据
  - 文档同步完成

## 9. 当前风险与后续冻结项

- `README / 开发启动手册 / 当前开发计划 / 统一接口文档` 若不同步，会造成阶段口径与验收口径漂移
- `wallet-service` 当前保留多链装配事实，不能把“仅配置限制 ETH 验证”误写成“已单链化”
- `market.price.tick` 属于高频 Kafka 链路，补失败重试专项时最容易影响 TP / 强平 / 浮盈亏链路
- `market.kline.update` 本轮新增 `trading` 侧消费，会带来幂等、重试、日志和文档联动风险
- `Swap / 隔夜利息` 继续冻结为待确认项，未确认前不得进入编码

## 10. 当前不建议抢跑的事项

- 直接进入 `Stage 6B / 6C / 7 / 7A`
- 在外部链节点与 Tiingo 真源仍未进入同一自动化用例前，把当前验证误写成“全平台联调完成”
- 在 `Swap` 规则未确认前编写隔夜利息实现
- 在 `Stage 6A` 未收口前把系统表述为“生产可用”或“可对外公测”
