# FalconX 当前开发计划

## 1. 项目定位与约束

- FalconX `v1` 采用 `GODSA`，一期固定服务为 `gateway / identity-service / market-service / trading-core-service / wallet-service`。
- `Stage 1-4` 解决服务边界、调用链和最小对外接口；`Stage 5` 起切入 owner MySQL、Flyway、MyBatis + XML、Redis、ClickHouse、Kafka 等真实基础设施。
- 当前计划仍以架构文档、开发启动手册、专题规范和仓库 `AGENTS.md` 为准；本文件只保留高信噪比进度总览。
- 当前系统仍不是生产可用状态：`Stage 6A / 6B / 6C / 7 / 7A` 的未完成项会直接影响外部联调、安全性和完整业务闭环。

## 2. 阶段完成概览

- `Stage 1` 已完成：共享模块、多模块父工程、5 个服务与 contract 基础结构已落地。
- `Stage 2A` 已完成：`market-service` 市场数据底座已落地。
- `Stage 2B` 已完成：`wallet-service` 钱包原始事实底座已落地。
- `Stage 3A` 已完成：`identity-service` 身份域基础闭环已落地。
- `Stage 3B` 已完成：`trading-core-service` 交易核心基础闭环已落地。
- `Stage 4` 已完成：`gateway` 与各服务的最小北向链路已打通。
- `Stage 5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、Redisson、ClickHouse、Kafka 已切到真实实现。
- `Stage 6A` 部分完成：`market-service` 已完成 Tiingo 真连接、owner 白名单热刷新、真实 K 线聚合；`wallet-service` 已进入 EVM 原生币最小真实扫块链路。
- `Stage 6B` 未完成：安全完整化未启动验收。
- `Stage 6C` 未完成：Jackson 3 迁移仍待单独启动。
- `Stage 7` 未完成：端到端业务闭环未完成。
- `Stage 7A` 未完成：逐仓模式改造未完成。

## 3. 当前真实能力摘要

### 项目级能力

- 多模块工程、owner 服务边界、统一错误码/响应结构、`traceId` 透传已稳定落地。
- 4 个 owner 服务均已启用 Flyway migration 和真实 MySQL 持久化。
- 业务查询链路已切到 `MyBatis + XML Mapper`，不再依赖正式实现路径上的 in-memory owner。
- Redis / Redisson、ClickHouse、Kafka 客户端与服务内投递链路已接入。

### 当前已落地的服务能力

- `identity-service`
  - 已提供 `register / login / refresh`。
  - Refresh Token 会话已持久化。
  - 已消费 `falconx.trading.deposit.credited` 推进用户激活。
- `market-service`
  - 已接入 Tiingo JDK WebSocket 真连接。
  - 平台放行范围来自 owner `t_symbol.status=1`，并支持运行时热刷新。
  - 已落地最新价缓存、`quote_tick` 写入、K 线收盘聚合、`GET /api/v1/market/quotes/{symbol}`。
  - 已具备 Tiingo crypto symbol 发现导入的最小能力，新增品种以 `status=2 suspended` 写入 owner 库。
- `wallet-service`
  - 已落地钱包地址分配和 owner 持久化。
  - 已具备 EVM 原生币最小真实扫块识别能力，依赖 `t_wallet_chain_cursor` 维护游标。
  - 已支持链上回滚窗口里的 reversal 观察。
  - 跨服务稳定原始交易主键已统一为 `walletTxId`。
- `trading-core-service`
  - 已落地业务入金入账、账户查询、市价开仓。
  - 已消费 `wallet.deposit.confirmed / reversed` 和 `market.price.tick`。
  - 账户查询已按最新行情快照动态计算 `unrealizedPnl`，并返回 `quoteStale`。
  - 持仓已支持 `takeProfitPrice / stopLossPrice` 字段存储，但触发执行仍未完成。

## 4. 当前未完成事项摘要

- `Stage 6A`
  - `wallet-service` 的 ERC20/代币精度解析、更多确认推进策略仍未完成。
  - 真实业务 Topic 联调、失败重试和跨服务连通性验证仍未完成。
  - Swap/隔夜利息结算仍未实现。
  - 若要对外提供实时订阅，北向 WebSocket 行情链路仍待按合同补齐。
- `Stage 6B`
  - RSA 密钥外部化仍未完成。
  - Token 黑名单、注册/登录限流、生产态安全配置仍未完成。
- `Stage 6C`
  - Jackson `3.x` 迁移未启动，当前仍运行在 `Jackson 2.21.2` 兼容路径。
- `Stage 7`
  - 手动平仓未实现。
  - TP/SL 自动触发、强平执行、负净值保护未实现。
  - 平仓/强平后的净敞口回补与完整对账链路未完成。
- `Stage 7A`
  - `margin_mode`、持仓终态字段、`trade_type`、追加逐仓保证金等 schema 与业务链路仍未完成。

## 5. 建议执行顺序

1. 先收口 `Stage 6A` 剩余真实外部联调项，确保行情、钱包、消息总线的跨服务验证闭环可复现。
2. 再推进 `Stage 6B` 安全完整化，补齐密钥管理、黑名单和限流，避免在生产态能力未补齐前扩大对外暴露面。
3. `Stage 6C` 作为独立技术专项处理，不与普通业务需求混做。
4. `Stage 7` 建议先做“手动平仓 + 平仓终态持久化”的最小闭环，再继续 TP/SL、强平和负净值保护。
5. `Stage 7A` 最后实施，基于 `Stage 7` 的平仓与风控闭环补齐 `ISOLATED` 模式。
