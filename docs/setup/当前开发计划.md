# FalconX 当前开发计划

## 1. 当前项目判断

- FalconX `v1` 采用 `GODSA`，一期固定服务为 `gateway / identity-service / market-service / trading-core-service / wallet-service`。
- `Stage 1-5` 已把 owner MySQL、Flyway、MyBatis + XML、Redis、ClickHouse、Kafka 和最小北向链路切到真实实现。
- 当前系统应表述为：**真实基础设施已接入、`Stage 7` 当前范围已验收、`Stage 6B` 当前范围已验收、且因 `Stage 6A / 7A` 仍有缺口而尚未达到生产可用口径的内部集成版本**。
- 当前系统仍不能表述为“生产可用”或“可安全对外公测”，因为 `Stage 6A / 7A` 的缺口仍会直接影响外部联调稳定性、交易安全边界和上线完整性。

## 2. 阶段完成概览

- `Stage 1` 已完成：共享模块、多模块父工程、5 个服务与 contract 基础结构已落地。
- `Stage 2A` 已完成：`market-service` 市场数据底座已落地。
- `Stage 2B` 已完成：`wallet-service` 钱包原始事实底座已落地。
- `Stage 3A` 已完成：`identity-service` 身份域基础闭环已落地。
- `Stage 3B` 已完成：`trading-core-service` 交易核心基础闭环已落地。
- `Stage 4` 已完成：`gateway` 与各服务的最小北向链路已打通。
- `Stage 5` 已完成：owner MySQL、Flyway、MyBatis + XML、Redis、Redisson、ClickHouse、Kafka 已切到真实实现。
- `Stage 6A` 部分完成：`market-service` 已完成 Tiingo 真连接、owner 白名单热刷新、真实 K 线聚合；`wallet-service` 已进入 EVM 原生币最小真实扫块链路。
- `Stage 6B` 当前范围已验收：密钥外置、Gateway 黑名单校验、`logout -> blacklist -> gateway reject` 闭环、认证入口限流、`/api/v1/trading/**` 每用户每秒 10 次限流、全局按 IP 每分钟 200 次兜底限流、登录失败锁定、注册限流、Gateway 熔断 / 超时 / fallback 已落地。
- `Stage 6C` 未完成：Jackson 3 迁移仍待单独启动。
- `Stage 7` 当前范围已验收：手动平仓、PATCH TP/SL、TP/SL 自动触发、强平执行、负净值保护、owner 级验证与代表性 gateway E2E 已收口；`net_exposure_usd / hedge_threshold_usd / hedge log / 告警` 这一组 B-book 风险可观测性基础已在当前范围落地，但这不改变系统整体仍未达到生产可用口径的事实。
- `Stage 7A` 未完成：逐仓增强能力未完成。

## 3. 当前真实能力摘要

### 项目级能力

- 多模块工程、owner 服务边界、统一错误码 / 响应结构、`traceId` 透传已稳定落地。
- 4 个 owner 服务均已启用 Flyway migration 和真实 MySQL 持久化。
- 业务查询链路已切到 `MyBatis + XML Mapper`，不再依赖正式实现路径上的 in-memory owner。
- Redis / Redisson、ClickHouse、Kafka 客户端与服务内投递链路已接入。

### 经文档与代码抽查确认的已落地能力

- `identity-service`
  - 已提供 `register / login / refresh / logout`。
  - Refresh Token 会话已持久化。
  - 已消费 `falconx.trading.deposit.credited` 推进用户激活。
  - RSA 密钥与数据库账号密码已改为环境变量注入。
  - 已落地登录失败锁定策略，并具备 Access Token 黑名单 Redis 存储能力；`POST /api/v1/auth/logout` 已可把当前 Access Token 的 `jti` 按剩余 TTL 写入黑名单。
  - 当前 `logout` 只吊销当前 Access Token，不引入新的 Refresh Token 主动撤销语义。
- `gateway`
  - 已建立 JWT 鉴权、`traceId` 透传、统一 fallback 控制器与下游路由熔断配置。
  - 已在鉴权阶段校验 Access Token `jti` 黑名单 Redis key。
  - 已对 `/api/v1/auth/login` 与 `/api/v1/auth/register` 落地每 IP 每分钟 20 次限流。
  - 已对 `/api/v1/trading/**` 落地每用户每秒 10 次限流，并对所有 `/api/v1/**` 请求落地每 IP 每分钟 200 次兜底限流。
  - 已为 identity / market / trading / wallet 路由补 `connect-timeout`、`response-timeout` 与 `CircuitBreaker`，下游超时或不可用时统一返回 `90002`。
- `market-service`
  - 已接入 Tiingo JDK WebSocket 真连接。
  - 平台放行范围来自 owner `t_symbol.status=1`，并支持运行时热刷新。
  - 已落地最新价缓存、`quote_tick` 写入、K 线收盘聚合、`GET /api/v1/market/quotes/{symbol}`。
  - 已具备 Tiingo crypto symbol 发现导入的最小能力，新增品种以 `status=2 suspended` 写入 owner 库。
- `wallet-service`
  - 已落地钱包地址分配和 owner 持久化。
  - 已具备 EVM 原生币与 ERC20 `Transfer` 最小真实扫块识别能力，依赖 `t_wallet_chain_cursor` 维护游标。
  - 已显式完成 `token decimals -> 业务金额` 换算，owner 持久化与 wallet 事件统一保留 `8` 位小数。
  - 已可通过确认窗口重扫把已识别原始交易稳定推进到 `DETECTED / CONFIRMING / CONFIRMED` 主链路。
  - 已支持链上回滚窗口里的 reversal 观察。
  - 跨服务稳定原始交易主键已统一为 `walletTxId`。
- `trading-core-service`
  - 已落地业务入金入账、账户查询、市价开仓和手动平仓最小链路。
  - 已具备 `wallet.deposit.confirmed / reversed` 与 `market.price.tick` 的正式消费入口；本轮 `Stage 7` 的直接代表性验证聚焦在 `wallet.deposit.confirmed` 与 `market.price.tick` 驱动的主链路，不包含 `wallet.deposit.reversed` 的代表性 E2E。
  - 账户查询已按最新行情快照动态计算 `unrealizedPnl`，并返回 `quoteStale`。
  - 已通过 `V5` schema 落地 `margin_mode / close_* / trade_type`，当前只真实支持 `margin_mode=isolated`。
  - 已支持 `PATCH /api/v1/trading/positions/{positionId}` 修改 `takeProfitPrice / stopLossPrice`，未传字段保持原值，显式 `null` 表示清空。
  - 已落地 TP/SL 自动触发、强平执行、负净值保护、`t_liquidation_log` 写入，以及 `trading.position.closed / trading.liquidation.executed` outbox 事件。
  - 已在开仓、平仓 / 强平和 fresh quote tick 三条链路刷新 `net_exposure_usd`，并按 `hedge_threshold_usd` 输出 `trading.risk.hedge.alert / recovered` 日志、写入 `t_hedge_log`，在超阈值时发布服务内 Spring Event stub；当前阶段不自动执行 A-book 对冲，也不新增 Kafka topic 契约。

## 4. CFD 视角下的当前缺口判断

从 CFD 平台可运营性的角度，当前最大的缺口已经不再是 owner 级交易风险闭环，而是下面 4 个跨服务与上线边界问题：

1. **全平台运行时联调仍不完整**
   - `wallet -> trading`、`trading -> identity` 两段 Kafka 入口，以及 `gateway + identity-service + trading-core-service + Kafka` 的代表性 E2E（`TC-E2E-001 / 010 / 011`）已经通过。
   - 但这不等于 `wallet-service` 真运行时监听、`market-service` 真运行时进程和更完整全平台链路已在同一 E2E 中验证完成。

2. **外部事实与基础设施联调仍不完整**
   - `wallet-service` 虽已补齐 EVM 最小 ERC20 / 精度换算 / 确认推进，但更完整链重组处理、更多链类型与生产级 metadata 治理仍未完成。
   - `market-service` 的 owner 数据、Tiingo 接入和 Redis 最新价已可支撑当前交易路径，但真运行时北向链路与更完整用户侧验证仍未纳入本轮 E2E。

3. **安全上线基线当前范围已验收，但不等于全平台生产可用**
   - 已形成事实的能力包括：密钥外置、`logout -> blacklist -> gateway reject`、认证入口限流、`/api/v1/trading/**` 每用户每秒 10 次限流、全局按 IP 每分钟 200 次兜底限流、登录失败锁定、注册限流、Gateway 熔断 / 超时 / fallback。
   - 当前边界：`logout` 只吊销当前 Access Token，不引入 Refresh Token 主动撤销；本轮验证仍以 `gateway + identity-service + trading-core-service + Kafka` 为主，不等于全平台真运行时统一验收。
   - 因此，安全基线的当前范围已闭合，但系统整体仍不能表述为“可直接对外公测”。

4. **用户侧实时与逐仓增强仍未完成**
   - 北向 WebSocket、更多用户视角查询、Swap / 隔夜利息结算、追加保证金和更完整 `ISOLATED` 增强仍未完成。
   - 这些能力不会阻塞 `Stage 7` 最小风险闭环，但会影响后续对外体验与运营完整性。

## 5. 后续优先级排序

默认按下面顺序推进，而不是按“谁更像完整功能”排序：

1. `Stage 6A` 剩余真实联调收口
2. 用户侧实时与运营完整性补齐
3. `Stage 7A` 逐仓增强
4. `Stage 6C` Jackson 3 独立技术专项

排序理由：

- `Stage 6B` 当前范围已验收，后续优先级可回到外部真运行时联调、用户侧实时能力与逐仓增强补齐。
- `Jackson 3` 属于重要但可隔离的技术专项，不应阻塞平仓 / 强平 / 风控闭环。
- 北向 WebSocket、更多链类型、更多产品完善度，优先级低于交易安全和安全基线。

## 6. 建议里程碑

### Milestone A：`Stage 6A` 收口

目标：

- 把真实外部源、业务 Topic 和跨服务追踪链先收口到“可稳定复现”

重点事项：

- 补完真实业务 Topic 联调、失败重试和跨服务连通性验证
- 明确 `walletTxId`、Outbox / Inbox、Redis 快照和 Kafka 消费链路的联通性验证

验收重点：

- `注册 -> 入金 -> 激活 -> 开仓` 主链路可稳定复现
- `wallet -> trading -> identity` 的事件追踪和幂等链路清晰可审计

### Milestone B：`Stage 7` 交易风险最小闭环

目标：

- 形成 FalconX 一期“可安全开仓、可安全平仓、可执行强平”的最小闭环
- 当前状态：`当前范围已验收`

重点事项：

- 手动平仓最小链路
  - 当前状态：`已完成`
  - 已落地：`POST /api/v1/trading/positions/{positionId}/close`、`V5` schema、非交易时段手动平仓放行、`t_account / t_ledger / t_position / t_trade / t_risk_exposure / t_outbox(trading.position.closed)` 同事务回补、事务提交后移除 `OpenPositionSnapshotStore`
- TP/SL 自动触发
  - 当前状态：`已完成`
- 强平候选识别与强平执行
  - 当前状态：`已完成`
- 平仓 / 强平后的 `t_risk_exposure` 同事务回补
  - 当前状态：`已完成`
- 负净值保护与极端行情兜底
  - 当前状态：`已完成`
- 以 `net_exposure_usd` 和对冲告警为代表的 B-book 风险可观测性基础
  - 当前状态：`已完成`
  - 当前口径：`net_exposure_usd`、`hedge_threshold_usd`、`t_hedge_log`、告警 / 恢复日志和服务内 Spring Event stub 已落地；当前阶段不自动执行 A-book 对冲，也不新增 Kafka topic 契约

验收重点：

- `注册 -> 入金 -> 激活 -> 开仓 -> 平仓 / TP / SL / 强平` 的代表性链路验证通过
- 终态持仓、账本、风险暴露和审计日志能形成完整对账链
- 当前状态：`当前范围已验收`
- 已通过证据：
  - owner 级实现与定向测试已覆盖：手动平仓、PATCH TP/SL、TP/SL 自动触发、强平执行、负净值保护、重复 tick 幂等、强平失败整笔回滚、Kafka price tick 切到自管执行器线程
  - owner 级实现与定向测试已覆盖：`net_exposure_usd`、`hedge_threshold_usd`、`t_hedge_log` 与超阈值 / 恢复日志
  - gateway 代表性 E2E 已通过：`TC-E2E-001`、`TC-E2E-010`、`TC-E2E-011`
- 边界说明：
  - 当前阶段只完成 B-book 风险观测、告警和日志留痕；真实 A-book 对冲执行仍属于后续专项
  - 本里程碑的 E2E 仍是 `gateway + identity-service + trading-core-service + Kafka` 范围，不包含 `wallet-service` 与 `market-service` 真运行时同时参测
  - 因 `Stage 6A / 6B / 7A` 仍有缺口，当前不能把系统表述为全平台生产可用

### Milestone C：`Stage 6B` 安全上线前基线

目标：

- 把当前开发态安全实现补齐到“可以准备外部测试”的最低水平
- 当前状态：`当前范围已验收`

重点事项：

- RSA 密钥外部化
- Token 黑名单
- 登录 / 注册限流与账户锁定
- Gateway 限流、熔断、超时与生产态安全配置

已形成事实的能力：

- RSA 密钥与数据库账号密码已外部化
- gateway 已在 JWT 校验阶段接入 Token 黑名单 Redis 检查
- `POST /api/v1/auth/logout` 已落地，当前 Access Token 可写入黑名单并被 gateway 拒绝复用
- `/api/v1/auth/login` 与 `/api/v1/auth/register` 已落地每 IP 每分钟 20 次认证入口限流
- `/api/v1/trading/**` 已落地每用户每秒 10 次限流
- 所有 `/api/v1/**` 已落地每 IP 每分钟 200 次全局兜底限流
- `identity-service` 已落地登录失败锁定
- gateway 已完成下游熔断 / 超时 / fallback

当前边界：

- 当前 `logout` 只吊销当前 Access Token，不引入 Refresh Token 主动撤销语义
- 当前范围验收不等于 `wallet-service` 与 `market-service` 真运行时已纳入同一 E2E
- 当前范围验收不等于全平台生产可用

验收重点：

- 认证、刷新、限流、黑名单、熔断和关键安全日志通过专项验证

### Milestone D：用户侧实时与运营完整性

目标：

- 在安全和风险闭环已形成后，再补对外体验与运营完整性

重点事项：

- 北向 WebSocket 行情订阅
- 订单 / 成交 / 账本等用户视角查询补齐
- Swap / 隔夜利息结算
- 更多运营告警与观测项

### Milestone E：`Stage 7A` 逐仓增强

目标：

- 在已有平仓 / 强平模板基础上，补齐显式 `ISOLATED` 能力增强

重点事项：

- `margin_mode` 对外显式化与查询回显
- 追加逐仓保证金接口与 `biz_type=10`
- 追加保证金后的强平价重算与风控联动

### Milestone F：`Stage 6C` Jackson 3 独立技术专项

目标：

- 在业务闭环和安全基线稳定后，独立完成 JSON 技术栈迁移

重点事项：

- 依赖 / BOM 收口
- Kafka / Redis / MyBatis JSON 兼容回归
- JWT / Gateway / Tiingo / Web3 协议解析回归

## 7. 当前不建议抢跑的事项

- 在 `Stage 6B` 当前范围虽已验收、但 `Stage 6A / 7A` 仍未收口时，把系统表述为“可对外公测”或“全平台生产可用”
- 在 `wallet-service` 与 `market-service` 真运行时未纳入同一 E2E 前，把当前代表性链路验证误写成“全平台联调完成”
- 在安全上线基线未形成前优先投入 `Jackson 3` 主迁移
- 在链上金额精度、确认推进和更完整链重组治理未收口前优先扩更多链类型或地址隐私增强
