# 开发启动手册

## 1. 目的

本文件只负责两件事：

- 说明后续实施顺序
- 给出每个阶段的最小产出和验收标准

它不是架构规则文档。

服务边界、数据库边界、接口规则、事件规则、日志规则均以对应专题文档为准。

## 2. 当前阶段

FalconX 当前已完成 `Stage 1` 到 `Stage 5`。

当前状态必须这样理解：

- 已完成的是“架构骨架、服务边界、最小调用链，以及 owner MySQL / Redis / ClickHouse / Kafka 的真实基础设施接入”
- 当前正式执行阶段固定为 `Stage 6A 收口专项`
- 尚未完成的是“Tiingo 外部真源与外部链节点真扫块的统一自动化验证、Tiingo 稳定成功建连证据补齐，以及 Swap/隔夜利息规则确认与实现”
- 当前实现仍然保留部分超前接口和代码事实，但这些事实不改变“当前正式阶段仍是 `Stage 6A`”的判断
- `Stage 6B / 6C / 7 / 7A` 当前全部冻结，待 `Stage 6A` 收口后再重新确认启动顺序
- 因此当前版本尚不可用于生产

当前仓库主要内容：

- 架构文档与治理规范
- 多模块服务骨架
- 数据库蓝图、SQL 蓝图和 owner 服务运行时 migration
- 基础设施编排
- Stage 1–5 的最小测试、接口骨架与真实持久化集成测试

## 4. 环境要求

- `JDK 25`
- `Maven 3.9+`
- `Docker / Docker Compose`

## 5. 文档冻结前置条件

开始写代码前，必须先确认下面几点已经冻结：

1. 服务边界以主架构文档为准  
2. 数据库 owner 以数据库设计和 SQL 蓝图为准  
3. API、事件、状态机、事务幂等规则不再口头讨论  
4. 模块分层已固定：`common / domain / infrastructure / contract / service`  

## 6. 当前验证基线

当前阶段最小验证应为：

1. 文档边界一致  
2. `mvn clean compile` 通过  
3. `mvn test` 通过  
4. `docker compose config` 通过  
5. 本地依赖可启动  
6. 当前阶段声明的主调用链可跑通  
7. 若阶段声称已完成真实基础设施接入，必须由真实 MySQL / Redis / ClickHouse / Kafka（视本阶段范围）集成测试覆盖  

补充要求：

- 阶段验收不能只写命令名，必须同时说明业务链路验证结论
- 若当前阶段仍基于内存仓储或 stub 依赖，必须显式声明，避免误解为真实基础设施已打通
- 阶段已经切换到真实基础设施的部分，也必须明确写出“哪些已经是真实接入、哪些仍是 stub”
- `Stage 5` 起本地开发必须先执行 `docker compose up -d` 拉起 Kafka（端口 `9092`），再启动业务服务；否则 `@KafkaListener` 注册失败会导致 Spring 上下文启动报错

## 7. 实施顺序

后续实施仍按下面的 `Stage` 边界管理，但本轮只执行 `Stage 6A 收口专项`。

固定口径：

- 当前正式阶段仍是 `Stage 6A`
- `Stage 6B / 6C / 7 / 7A` 本轮全部不启动、不验收
- 只有在 `Stage 6A` 收口并完成文档同步后，才重新确认后续阶段顺序

### Stage 6A：外部源与消息总线真接入

目标：

- 打通真实报价源、真实链节点和真实 Kafka

本阶段应完成：

1. `market-service` 接入 Tiingo WebSocket 真连接、断线重连与 stale 判定  
2. `wallet-service` 接入至少一条真实链 RPC，并以配置限制 `ETH` 作为本轮唯一验收链路
3. 真实业务 Topic 联调与失败重试验证，包含 `market.price.tick` 的 Kafka 入口失败重试专项
4. Outbox / Inbox 与消息总线联通  
5. `market.kline.update` 在 `trading-core-service` 侧形成正式消费链路
6. 真实 K 线聚合落 ClickHouse

本阶段产出：

- 真实报价进入系统
- 真实链上事件进入系统
- 低频关键事件通过真实 Kafka 流转
- `wallet-service` 当前前期测试节点固定补充为：
  - `Ethereum Mainnet`：`wss://eth-mainnet.g.alchemy.com/v2/j6-5C7HK6dXPSSAyrZZqq`
  - `Ethereum Sepolia`：`wss://eth-sepolia.g.alchemy.com/v2/j6-5C7HK6dXPSSAyrZZqq`
  - `Ethereum Hoodi`：`wss://eth-hoodi.g.alchemy.com/v2/j6-5C7HK6dXPSSAyrZZqq`
- 现阶段只允许通过 `falconx.wallet.chains.eth.rpc-url` 在上述 3 条测试端点中切换其一，
  不新增新的链类型、schema 字段或独立服务配置模型
- 本地联调时可通过启动参数或 IDE 配置覆盖，例如：
  - `--falconx.wallet.chains.eth.rpc-url=wss://eth-sepolia.g.alchemy.com/v2/j6-5C7HK6dXPSSAyrZZqq`
- 当前已完成的最小真实链路是：
  - `wallet-service` 当前代码仍保留多链装配事实，但本轮验收只按配置限制 `ETH` 单链验证
  - 最新链头会回写到 owner `t_wallet_chain_cursor`
  - 监听器会按 owner 游标回扫最近确认窗口区块
  - 每轮扫描会先批量预载平台地址快照，避免按交易逐条查 owner 库
  - 命中平台地址的原生币转账会生成 `ObservedDepositTransaction`
  - 已确认原生币入金若在重扫窗口中从当前 canonical block 集合里消失，会生成 reversal 观察结果
  - 入金 `amount` 的口径已冻结为“按 token decimals 归一化后的业务金额，统一保留 8 位小数”，后续 ERC20 / SPL 解析必须先显式换算再进入 owner 持久化与事件
  - wallet -> trading 的跨服务唯一业务主键当前已切换为 `walletTxId`
  - `wallet-service` 当前会通过 `(chain, tx_hash, log_index)` 拆分原始链事实，再由 `walletTxId` 对外输出稳定事件标识
  - `trading-core-service` 的业务入账与回滚幂等当前已统一切到 `walletTxId`
  - 节点短时不可用时记录告警并等待下一轮轮询，不把整个服务启动链路升级成失败
- 当前仍未完成：
  - Tiingo 外部真源与外部链节点真扫块纳入同一自动化用例
  - Tiingo 稳定成功建连并持续收流的直接证据
  - `Swap / 隔夜利息` 的费率公式、结算时点、owner 读取路径与幂等规则确认

本阶段验收：

- 真实外部源到系统内部事件链路验证通过
- 失败 / 重试 / 超时路径测试通过
- 关键外部调用日志符合规范
