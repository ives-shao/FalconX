## 场景3：我要修复问题清单中的问题

### 第一步：查询问题状态

```bash
# 快速查询问题状态
grep "FX-XXX" docs/process/统一问题清单.md -A 5

# 查看问题优先级
grep "优先级" docs/process/统一问题清单.md | grep "FX-XXX"

# 查看问题状态
grep "当前状态" docs/process/统一问题清单.md | grep "FX-XXX"
```

### 第二步：根据状态决定行动

| 问题状态 | 行动 |
|---------|------|
| 待确认 | 先核对问题是否真实存在 |
| 待修复 | 检查代码，确认后修复 |
| 修复中 | 继续修复或接手 |
| 已完成 | 只更新文档，不改代码 |
| 不成立 | 确认后标记，不改代码 |
| 后续阶段处理 | 不在当前阶段处理 |

### 第三步：修复流程

```
1. 确认问题真实存在
   ↓
2. 读取相关规范文档（如果涉及架构/安全/事务）
   ↓
3. 修复代码
   ↓
4. 添加/更新测试
   ↓
5. 更新问题清单状态
   ↓
6. 更新相关文档（如果需要）
   ↓
7. 提交Git回滚点
```

## 场景4：我要修改数据库

### 必须遵守的规则

🔴 **强制：**
1. 必须先更新数据库设计文档
2. 必须使用Flyway migration
3. 必须遵守owner边界
4. 必须处理checksum影响

### 修改流程

```
1. 更新 docs/database/falconx一期数据库设计.md
   ↓
2. 创建新的migration文件（V{n}__description.sql）
   ↓
3. 更新对应的Entity/Mapper/XML
   ↓
4. 添加测试
   ↓
5. 验证migration执行
   ↓
6. 提交Git回滚点
```

### 特殊情况：修改基线migration

⚠️ **警告：修改V1/V2等基线脚本需要特别注意**

```
1. 必须处理Flyway checksum冲突
   ↓
2. 选择以下方案之一：
   - 使用新的隔离测试库
   - 执行Flyway repair（需要授权）
   ↓
3. 更新所有相关文档
   ↓
4. 通知团队成员
```

## 场景5：我要修改接口

### REST接口修改

| 修改类型 | 是否破坏性变更 | 需要做什么 |
|---------|--------------|-----------|
| 新增字段 | 否 | 更新统一接口文档 |
| 删除字段 | 是 | 版本升级或兼容处理 |
| 修改字段类型 | 是 | 版本升级或兼容处理 |
| 新增接口 | 否 | 更新统一接口文档 |
| 修改路径 | 是 | 版本升级 |

### 必须更新的文档

🔴 **强制更新：**
- `docs/api/FalconX统一接口文档.md`

🟡 **建议更新：**
- `docs/api/REST接口规范.md`（如果涉及新的规范）

### 接口文档必须包含

- [ ] 接口路径和方法
- [ ] 请求参数/请求体
- [ ] 响应结构

- [ ] 认证要求
- [ ] 测试结论

## 场景7：我要提交代码

### 提交前检查清单

#### 🔴 强制检查

- [ ] 是否破坏了服务owner边界？
- [ ] 是否修改了数据库schema？如果是，是否更新了文档？
- [ ] 是否修改了接口契约？如果是，是否更新了统一接口文档？
- [ ] 是否修改了事件payload？如果是，是否更新了Kafka事件规范？
- [ ] 是否涉及安全、事务、幂等？如果是，是否符合相关规范？
- [ ] 是否有明显的并发问题、资源泄漏、SQL注入风险？
- [ ] 是否生成了Git回滚点？
- [ ] 是否在dev分支？

#### 🟡 建议检查

- [ ] 是否有单元测试或集成测试？
- [ ] 是否有必要的注释和文档？
- [ ] 是否遵循了命名规范？
- [ ] 是否有重复代码可以复用？

#### 🟢 可选检查

- [ ] 是否使用了JDK 25新特性？
- [ ] 是否可以进一步优化性能？
- [ ] 是否可以简化代码逻辑？

### Git提交规范

```bash
# 1. 确保在dev分支
git branch --show-current  # 应该显示 dev

# 2. 查看修改内容
git status
git diff

# 3. 添加修改
git add .

# 4. 提交（使用清晰的commit message）
git commit -m "feat: add user activation logic

- Implement user status transition from PENDING_DEPOSIT to ACTIVE
- Add integration test for activation flow
- Update identity service documentation

Closes: FX-022"

# 5. 推送到远程（如果用户要求）
git push origin dev
```

## 场景8：我要处理安全相关代码

### 安全规则优先级

🔴 **P0 - 必须立即处理：**
- 明文存储密钥
- SQL注入风险
- 无限流/无熔断
- 无幂等保护

🟡 **P1 - 应尽快处理：**
- 密码强度不足
- Token有效期过长
- 无邮箱验证
- 无登录失败限制

🟢 **P2 - 优化项：**
- 日志优化
- 性能优化
- 用户体验优化

### 安全修改必须遵守

- [ ] 不得在日志中打印敏感信息
- [ ] 不得在配置文件中明文存储密钥（生产环境）
- [ ] 必须实现基础限流
- [ ] 必须实现幂等保护
- [ ] 必须更新安全规范文档

## 场景9：我要处理并发问题

### 常见并发场景

| 场景 | 解决方案 | 规则 |
|------|---------|------|
| 账户余额扣减 | SELECT ... FOR UPDATE | 🔴 强制 |
| 订单幂等 | 唯一约束 + 幂等键 | 🔴 强制 |
| 持仓快照更新 | afterCommit回调 | 🔴 强制 |
| Refresh Token轮换 | CAS + 行锁 | 🔴 强制 |
| 地址分配 | 锁定查询 + 重试 | 🔴 强制 |

### 并发测试要求

🔴 **强制测试：**
- 账户并发扣款
- 订单并发提交
- Token并发刷新

🟡 **建议测试：**
- 持仓并发更新
- 地址并发分配

## 场景10：我要处理性能问题

### 性能优化规则

| 优化类型 | 规则 | 注意事项 |
|---------|------|---------|
| 批量写入 | 🟡 建议 | 不破坏事务语义 |
| 缓存优化 | 🟡 建议 | 明确TTL和刷新策略 |
| 索引优化 | 🟡 建议 | 更新数据库文档 |
| 查询优化 | 🟡 建议 | 保持SQL在XML中 |
| 异步处理 | 🟡 建议 | 不影响业务一致性 |

### 性能优化必须保证

- [ ] 不破坏事务边界
- [ ] 不破坏幂等逻辑
- [ ] 不影响数据一致性
- [ ] 有性能测试验证
