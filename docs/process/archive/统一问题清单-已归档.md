## [FX-055] Stage 7 与 Stage 7A 的职责重叠，导致交易风险闭环顺序表达冲突

- 优先级：`P1`
- 当前状态：`已完成`

旧文档口径中：

- `Stage 7` 要求补齐 TP/SL 自动触发、强平闭环和负净值保护

从 CFD 平台风险控制视角看，这会把最关键的“可安全平仓 / 可安全强平”能力排到过晚位置。

### 涉及范围

  - `docs/setup/当前开发计划.md`
  - `docs/process/逐仓模式改造方案.md`

### 期望修复

统一口径：

- `Stage 7` 负责交易风险最小闭环，并吸收手动平仓 / 强平基础能力所需的最小 schema 前置
- `Stage 7A` 改为在 `Stage 7` 基础上补齐显式 `ISOLATED` 增强，如追加保证金和强平价重算
- `Stage 6C` 保持独立技术专项，不再作为 `Stage 7 / 7A` 的默认阻塞门槛

### 核对结论

- 是否真实存在：`是`
- 结论说明：核对 `开发启动手册`、`当前开发计划` 与 `逐仓模式改造方案` 后确认旧表述确实存在阶段职责重叠和先后依赖矛盾。

  - `当前开发计划` 已按 CFD 风险优先级重排后续里程碑

## [FX-051] wallet-service 无法识别已确认原生币入金在确认窗口内消失的回滚场景

- 优先级：`P1`
- 当前状态：`已完成`

### 问题描述

`wallet-service` 已能按 owner 游标回扫最近确认窗口区块，并识别命中平台地址的原生币入金。
但旧实现只会处理“窗口里当前还能看到的交易”，不会把“曾经已落地且已确认、现在却从当前 canonical block 集合里消失”的交易识别为回滚候选。
这会让 `CONFIRMED -> REVERSED` 的最小原生币回滚链路缺少监听入口。

### 涉及范围

- 文档：
  - `docs/setup/当前开发计划.md`
  - `docs/setup/开发启动手册.md`
  - `falconx-wallet-service/README.md`
- 模块：
  - `falconx-wallet-service`
- 文件：
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/listener/Web3jChainDepositListener.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/WalletDepositTransactionRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/MybatisWalletDepositTransactionRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/mapper/WalletDepositTransactionMapper.java`
  - `falconx-wallet-service/src/main/resources/mapper/wallet/WalletDepositTransactionMapper.xml`
  - `falconx-wallet-service/src/test/java/com/falconx/wallet/listener/Web3jChainDepositListenerTests.java`

### 期望修复

让 EVM listener 在确认窗口回扫时，同时加载该窗口内 owner 已落地的原始交易；
若已 `CONFIRMED` 的原生币入金在当前 canonical block 集合里消失，则生成 `reversed=true` 的 `ObservedDepositTransaction`，
交由应用层按既有状态机推进 `CONFIRMED -> REVERSED`。

### 核对结论

- 是否真实存在：`是`
- 结论说明：核对代码确认旧实现只处理“当前扫到的链上交易”，没有对已落地窗口交易做缺失对账，因此回滚识别入口缺失。

### 修复记录

- 修复内容：
  - `WalletDepositTransactionRepository` 新增按链和区块范围加载 owner 原始交易的查询能力
  - `Web3jChainDepositListener` 现在会在确认窗口内同时加载 owner 已跟踪交易，并按 `(txHash, logIndex)` 收集当前 canonical block 集合中的原始链事实
  - 对已 `CONFIRMED` 且在当前窗口彻底消失的原始入金（原生币 `logIndex=0` / ERC20 `Transfer logIndex>0`），会生成 `reversed=true` 的观察结果交给应用层
  - listener 后续补齐 ERC20 最小解析后，reversal 对账也保持与 `(chain, tx_hash, log_index)` 一致，不再把同交易多条 transfer 混成同一链事实
- 验证结果：
  - `mvn -pl falconx-wallet-service -am test`
  - `mvn clean compile`

## [FX-050] wallet-service EVM 扫块按交易逐条查 owner 地址，形成 N+1 查询压力

- 优先级：`P1`
- 当前状态：`已完成`

`walletAddressRepository.findByChainAndAddress(...)`。当扫描窗口变大或链上交易密度上升时，
这会把 listener 放大成“每条交易一次 owner 库查询”的 N+1 模式，和后续生产态性能要求不匹配。

  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/WalletAddressRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/MybatisWalletAddressRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/mapper/WalletAddressMapper.java`
  - `falconx-wallet-service/src/main/resources/mapper/wallet/WalletAddressMapper.xml`
  - `falconx-wallet-service/src/test/java/com/falconx/wallet/listener/Web3jChainDepositListenerTests.java`

### 期望修复

让 EVM 监听器在每轮扫描开始时一次性批量加载当前链的启用平台地址快照，
后续按交易只做内存匹配，不再对每笔链上交易逐条查 owner 数据库。

### 核对结论

- 是否真实存在：`是`
- 结论说明：核对监听器实现确认此前按交易逐条调用地址 Repository，存在明显的 N+1 查询路径。

### 修复记录

- 修复内容：
  - `WalletAddressRepository` 新增按链批量加载启用地址集合的方法
  - MyBatis XML 已补齐对应批量查询 SQL
  - `Web3jChainDepositListener` 每轮同步时先预载地址快照，再按交易做内存匹配
  - 地址匹配统一做小写归一化，降低 EVM checksum 大小写差异带来的误判风险
- 验证结果：
  - `mvn -pl falconx-wallet-service -am test`
  - `mvn clean compile`

## [FX-011] `t_risk_exposure` UPSERT `net_exposure` double-count 计算 bug

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

`TradingRiskExposureMapper.xml` 中 `applyLongDelta` 和 `applyShortDelta` 的 `ON DUPLICATE KEY UPDATE` 语句存在财务计算错误。

MySQL `ON DUPLICATE KEY UPDATE` 中，同一 `UPDATE` 子句内，**后续列引用前面已更新过的列时得到的是更新后的值**。当前写法导致 `net_exposure` 每次被多加（或多减）一个 delta：

```sql
-- applyLongDelta（有问题）
total_long_qty = total_long_qty + VALUES(total_long_qty),          -- ① total_long_qty 已变为 old+delta
net_exposure   = (total_long_qty + VALUES(total_long_qty)) - total_short_qty  -- ② 此处 total_long_qty=old+delta, VALUES=delta
-- 实际结果：net_exposure = old_long + 2×delta - old_short  ← 多加了一个 delta
```

同理 `applyShortDelta` 也有对称的 double-count 问题。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：`falconx-trading-core-service/src/main/resources/mapper/trading/TradingRiskExposureMapper.xml`（第 33、53 行）

### 期望修复

`ON DUPLICATE KEY UPDATE` 内 `net_exposure` 直接引用**已更新后**的方向字段即可，不再重复加 `VALUES()`：

```sql
-- applyLongDelta
total_long_qty = total_long_qty + VALUES(total_long_qty),
net_exposure   = total_long_qty - total_short_qty,   -- total_long_qty 已是更新后的值

-- applyShortDelta
total_short_qty = total_short_qty + VALUES(total_short_qty),
net_exposure    = total_long_qty - total_short_qty,  -- total_short_qty 已是更新后的值
```

### 核对结论

- 是否真实存在：`是`
- 结论说明：直接读取 `TradingRiskExposureMapper.xml` 第 32-34 行和第 51-53 行，与 MySQL `ON DUPLICATE KEY UPDATE` 列求值语义对照，确认每次开仓后 `net_exposure` 偏差为 `+delta`（多头方向）或 `-delta`（空头方向）。

### 修复记录

- 修复内容：
  - `TradingRiskExposureMapper.xml` 的多头/空头 UPSERT 已改为基于更新后的 `total_long_qty / total_short_qty` 直接重算 `net_exposure`
  - 去掉了重复叠加 `VALUES(...)` 导致的净敞口偏移
- 验证结果：

## [FX-036] `wallet.deposit.reversed` 在 owner 入金事实不存在时被错误提前确认

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

`TradingDepositReversalApplicationService` 旧实现存在一个真实幂等缺陷：
当 `wallet.deposit.reversed` 先到、而 `walletTxId` 对应的交易域入金事实尚不存在时，
代码会先把该 `eventId` 写入成功 `t_inbox`，随后仅记录告警并返回。

这会违反《事务与幂等规范》中“owner 业务对象不存在时必须消费失败并重试，不能提前标记成功”的要求。
一旦后续 `wallet.deposit.confirmed` 才到达，之前那条 `reversed` 可能已经被错误吞掉，导致业务入金事实无法被及时回滚。

### 涉及范围

- 文档：
  - `docs/architecture/事务与幂等规范.md`
- 模块：
  - `falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingDepositReversalApplicationService.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/TradingKafkaWalletDepositIntegrationTests.java`

### 期望修复

当 `wallet.deposit.reversed` 先到而 owner 入金事实不存在时：
- 不允许写成功 `t_inbox`
- 必须抛错，让 Kafka 保持失败并走重试
- 待 `wallet.deposit.confirmed` 完成入账后，之前失败的 `reversed` 应可在重试中成功完成业务回滚

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮通过代码核对与真实 Kafka 入口集成测试确认，该问题真实存在；其触发条件正是 `reversed` 先到、`confirmed` 后到的乱序场景。

### 修复记录

- 修复内容：
  - `TradingDepositReversalApplicationService` 已改为：当 `walletTxId` 对应交易域入金事实不存在时直接抛错，不再写成功 `t_inbox`
  - 新增真实 Kafka 入口集成测试，已验证：
    - `wallet.deposit.confirmed` 真消费并完成业务入账
    - duplicate confirmed 不重复入账
    - `wallet.deposit.reversed` 真消费并完成业务回滚
    - `reversed` 先到时初始阶段不写成功 `t_inbox`；待 `confirmed` 到达后，失败的 `reversed` 会在重试中成功完成 reversal
- 验证结果：
  - `wallet.deposit.confirmed` 进入真实 Kafka 入口后，会由 `TradingKafkaEventListener` 真消费，并写入 `t_inbox / t_deposit / t_outbox / t_account`，同时产出 `trading.deposit.credited` Kafka 消息。
  - duplicate `wallet.deposit.confirmed` 不会重复入账；`walletTxId` 业务幂等成立，且不会重复产出第二条 `trading.deposit.credited` 消息。
  - `wallet.deposit.reversed` 会把已入账的 `deposit` 事实回滚，`t_deposit.status` 进入 `REVERSED`，账户余额回到 `0.00000000`。
  - `wallet.deposit.reversed` 先到时，不会提前写成功 `t_inbox`；待 `wallet.deposit.confirmed` 到达后，之前失败的 `reversed` 会在重试中完成最终回滚。
  - 已执行命令：
    - `mvn -pl falconx-trading-core-service -am test -Dsurefire.failIfNoSpecifiedTests=false -Dtest=TradingKafkaWalletDepositIntegrationTests`
    - `mvn -pl falconx-trading-core-service -am test -Dsurefire.failIfNoSpecifiedTests=false -Dtest=TradingKafkaWalletDepositIntegrationTests,TradingPersistenceIntegrationTests,TradingKafkaEventListenerTests`

## [FX-038] 手动平仓在账户缺失时会 fail-open 隐式补建 `t_account`

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

Stage 7 第一轮手动平仓实现沿用了 `TradingAccountService.getOrCreateAccountForUpdate()` 的“读不到就补建”语义。
这导致当结算路径中的真实交易账户缺失，或同一用户仅存在非结算币种账户时，手动平仓不会立即失败，
而是可能隐式补建一条新的 `t_account`，随后继续推进平仓结算。

这会带来两个严重后果：
- 内部 owner 数据不一致被错误放大成新的账户事实，而不是让事务失败回滚
- 后续可能生成错误的 `margin_used` 快照、账本、平仓成交或 `trading.position.closed` outbox

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/TradingAccountService.java`
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/impl/DefaultTradingAccountService.java`
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingPositionCloseApplicationService.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/controller/TradingControllerIntegrationTests.java`

### 期望修复

手动平仓只能结算已存在的真实交易账户：
- 禁止通过 `getOrCreateAccountForUpdate()` 或任何等价逻辑在平仓路径隐式补建账户
- 若账户缺失或币种不匹配，必须让整笔平仓失败并回滚
- 不允许生成新的 `t_account`、`t_trade`、`t_ledger.biz_type=8` 或 `trading.position.closed`

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮代码核对确认，旧实现的平仓结算路径直接调用 `getOrCreateAccountForUpdate()`，真实存在 fail-open 风险。

### 修复记录

- 修复内容：
  - `TradingAccountService` 已新增“仅获取已存在账户并加锁”的正式能力
  - `DefaultTradingAccountService.settleClosedPosition(...)` 已改为只接受已存在且已加锁的账户对象，不再自行补建账户
  - `TradingPositionCloseApplicationService` 已在写入账本、成交和 outbox 之前先锁定真实结算账户；若账户缺失或币种不匹配，直接走既有 500 回滚路径
- 验证结果：
  - `TradingControllerIntegrationTests.shouldReturnInternalErrorWithoutRecreatingSettlementAccountWhenCloseAccountIsMissing` 已验证：
    - 删除结算账户后执行手动平仓，HTTP 返回 `500 / 90001`
    - `t_account` 不会被重新创建
    - 原持仓保持 `OPEN`
    - 不会新增 `t_trade.trade_type=2`、`t_ledger.biz_type=8` 或 `trading.position.closed`
