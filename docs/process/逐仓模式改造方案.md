# 逐仓模式（Isolated Margin）改造方案

> 状态：**V2 修订版（评审合并后，可实施，待逐条确认）**
> 目标：在现有 GODSA 架构与当前账户语义不被破坏的前提下，以最小改造量补齐完整的逐仓保证金模式。
> 范围：本次只落 `ISOLATED`，`CROSS` 仅保留字段与错误码预留，不进入实现。
> 当前实施口径补充：
> - `Step 1-5` 属于 `Stage 7` 交易风险闭环的前置工作包
> - `Step 6-7` 才属于 `Stage 7A` 的逐仓增强工作包
> - 默认 `margin_mode=isolated` 的最小 schema 前置，不应再被排到手动平仓 / 强平基础能力之后

---

## 一、改造背景与现状分析

### 1.1 当前模型定性

| 维度 | 当前实现 | 结论 |
|------|---------|------|
| 强平价公式 | `entry × (1 ± 1/leverage ∓ mmRate)` | 仅适用于"按初始杠杆换算逐仓保证金"的开仓场景，追加保证金后必须重构 |
| 持仓 margin 字段 | `t_position.margin` 已存在 | 逐仓核心基础具备 |
| 强平触发 | **存根，未实现** | `QuoteDrivenEngine` 当前只写价格快照，不触发业务 |
| 强平执行 | **不存在** | 需新增 `LiquidationExecutionService` |
| 平仓链路 | **不存在** | 当前只有开仓，连手动平仓都没做 |
| `margin_mode` 字段 | **不存在** | schema 还不能显式区分 `isolated / cross` |
| 追加保证金 | **不存在** | 接口、账务、计算和日志都缺失 |

### 1.2 方案前置的真实 Schema 缺口

> 重要：下面这些字段在当前真实 Flyway `V1 / V2 / V3` 中仍未落地。若不先补齐，本方案从手动平仓开始的第一次 `UPDATE t_position / INSERT t_trade` 就会直接失败。

| 字段 | 当前真实 V1/V2/V3 | V4 是否必须补 | 说明 |
|------|------------------|-------------|------|
| `t_position.close_price` | 不存在 | **是** | 终态持仓必须记录平仓价 |
| `t_position.close_reason` | 不存在 | **是** | 一次冻结完整枚举：manual/tp/sl/liquidation |
| `t_position.realized_pnl` | 不存在（只在 `t_trade`） | **是** | 终态持仓需要直接回显已实现盈亏 |
| `t_position.closed_at` | 不存在 | **是** | 终态时间必须可追溯 |
| `t_trade.trade_type` | 不存在 | **是** | 开仓/平仓/强平必须显式区分 |
| `t_position.margin_mode` | 不存在 | **是** | 本方案新增，当前只允许 `isolated` |

### 1.3 当前架构的硬约束

1. **账户语义冻结，不可变更**
   - `balance`：账户总现金余额
   - `frozen`：预留未确认金额
   - `margin_used`：已确认占用保证金
   - `available = balance - frozen - margin_used`
   - 逐仓改造不能把 `balance` 错当成"已扣除保证金后的余额"

2. **`t_account` 一用户一账户**
   - `user_id` 唯一约束
   - 存在 `version` 乐观锁字段，但本方案统一改用 `SELECT FOR UPDATE`（避免强平场景下乐观锁冲突重试放大复杂度）

3. **owner 持久化必须走 MyBatis + XML**，禁止字符串 SQL

4. **Flyway 迁移真实目录**：`falconx-trading-core-service/src/main/resources/db/migration/`
   - 当前已存在 `V1 / V2 / V3`，本次使用 `V4`
   - 脚本中**不要**写 `USE falconx_trading`
   - `docs/sql/*.sql` 是蓝图，不参与执行

### 1.4 改造核心路径

```text
Step 0：文档先行 —— 先修正本方案、蓝图与全局规范的一致性
   ↓
Step 1：V4 Schema 一次性合并（margin_mode + 平仓字段 + trade_type + biz_type 扩展）
   ↓
Step 2：领域模型、Mapper XML、下单请求同步
   ↓
Step 3：手动平仓最小链路（作为强平的账务模板）
   ↓
Step 4：QuoteDrivenEngine 强平候选识别 + in-flight 去重 + 重启恢复
   ↓
Step 5：LiquidationExecutionService（复用平仓模板 + 负净值兜底）
   ↓
Step 6：追加保证金接口 + 强平价重算
   ↓
Step 7：文档与测试同步
```

## 二、Schema 变更（Step 1：V4 合并迁移）

### 2.1 变更目标

V4 作为"平仓字段一次性冻结 + 逐仓模式"的合并迁移：

1. 补齐平仓字段：`close_price / close_reason / realized_pnl / closed_at`
2. 补齐 `t_trade.trade_type`
3. 新增 `t_position.margin_mode`
4. 扩充 `t_ledger.biz_type` 至 10 个值

### 2.2 真实迁移路径

- 蓝图：`docs/sql/V1__init_schema.sql`（同步更新，非可执行）
- Flyway：`falconx-trading-core-service/src/main/resources/db/migration/V4__isolated_margin_and_close_fields.sql`

### 2.3 V4 SQL

```sql
-- 1) 平仓字段一次性冻结
ALTER TABLE t_position
    ADD COLUMN close_price   DECIMAL(24,8) NULL
        COMMENT '平仓价（仅终态持仓有值）' AFTER stop_loss_price,
    ADD COLUMN close_reason  TINYINT       NULL
        COMMENT '1=manual,2=tp,3=sl,4=liquidation' AFTER close_price,
    ADD COLUMN realized_pnl  DECIMAL(24,8) NULL
        COMMENT '已实现盈亏（仅终态持仓有值）' AFTER close_reason,
    ADD COLUMN closed_at     DATETIME(3)   NULL
        COMMENT '终态时间' AFTER realized_pnl;

-- 2) t_trade.trade_type
ALTER TABLE t_trade
    ADD COLUMN trade_type TINYINT NOT NULL DEFAULT 1
        COMMENT '1=open,2=close,3=liquidation' AFTER side;

-- 3) 逐仓模式字段
ALTER TABLE t_position
    ADD COLUMN margin_mode TINYINT NOT NULL DEFAULT 2
        COMMENT '1=cross,2=isolated' AFTER leverage;

-- 4) 账本类型扩展
ALTER TABLE t_ledger
    MODIFY COLUMN biz_type TINYINT NOT NULL
        COMMENT '1=deposit_credit,2=deposit_reversal,3=margin_reserved,4=fee_charged,5=margin_confirmed,6=swap_charge,7=swap_income,8=realized_pnl,9=liquidation_pnl,10=isolated_margin_supplement';
```

### 2.4 约束

- `margin_mode` 默认值固定为 `2=isolated`
- 一期应用层只允许写入 `2`；`CROSS` 在 DTO 反序列化层即拒绝
- `close_reason = 4` 用于强平终态，与手动平仓 / TP / SL 并列
- `t_ledger.biz_type=9` 使用中性命名 `liquidation_pnl`，`amount` 允许正负值，避免与“强平一定亏损”的命名冲突

### 2.5 账本 `biz_type=8` 与 `biz_type=9` 的区分维度

| biz_type | 枚举名 | 触发场景 | 对应 `close_reason` |
|----------|--------|---------|---------------------|
| 8 | `realized_pnl` | 手动平仓 / TP / SL 触发的平仓 | 1 / 2 / 3 |
| 9 | `liquidation_pnl` | QuoteDrivenEngine 驱动的强平 | 4 |

**关键规则：**

- 两者的 `amount` 语义完全相同：均为 `realizedPnl`，允许正负值
- `t_position.close_reason` 是**唯一权威来源**；`biz_type` 必须按 `close_reason` 派生：
  - `close_reason ∈ {1,2,3}` → `biz_type=8`
  - `close_reason = 4` → `biz_type=9`
- 区分维度仅用于审计分类：
  - `biz_type=9` 汇总反映平台风控有效性与负净值兜底总额
  - `biz_type=8` 汇总反映用户主动操盘与 TP/SL 策略表现

> 该规则需在 `docs/architecture/事务与幂等规范.md` 平仓 / 强平事务模板章节补齐说明。

## 四、手动平仓最小链路（Step 3，强平前置）

### 4.1 为什么先做手动平仓

强平执行 = 手动平仓 + 负净值兜底 + in-flight 去重。若直接从零造强平，账务事务模板没有参照。先打通手动平仓：

- 跑通开仓 → 平仓的账务闭环
- 固化账务模板（盈亏计算、margin 回补、ledger 快照、`t_risk_exposure` 回补、Outbox）
- 强平仅在此基础上增加"负净值兜底 + 二次校验 + in-flight"三件事

### 4.2 REST 接口

```text
POST /api/v1/trading/positions/{positionId}/close
Authorization: Bearer <accessToken>
```

### 4.3 错误码

| 错误码 | 场景 |
|--------|------|
| `40004` | positionId 不存在或不属于当前用户 |
| `40007` | 持仓已关闭或已强平 |
| `30002` | 行情 stale / 价格源断连，无法安全平仓 |
| `30003` | 无可用报价，无法计算平仓价 |

### 4.4 执行逻辑（同一 `@Transactional`）

```text
closePosition(userId, positionId):

  ① SELECT t_position WHERE id=? AND user_id=? FOR UPDATE
     status != OPEN → 幂等返回 40007

  ② SELECT t_account WHERE user_id=? FOR UPDATE

  ③ 读取 markPrice
     - 从 Redis 最新价读取
     - stale 判定统一复用全局冻结规则：`当前时间 - quote.ts > 5s` 即视为 `STALE`
       · 非交易时段 Tiingo 停推价格会被该阈值自然识别为 stale
       · `stale=true` 返回 `30002`（Price Source Stale Or Disconnected）
       · Redis 无最新价返回 `30003`（Quote Not Available）
     - **不**做交易时间校验；非交易时段也允许手动平仓

  ④ 计算
     direction   = side==BUY ? +1 : -1
     realizedPnl = (markPrice - entryPrice) × quantity × direction

  ⑤ 更新 t_account
     balance     += realizedPnl
     margin_used -= position.margin
     frozen       不变
     version     += 1

  ⑥ 写 t_ledger（biz_type=8 realized_pnl）
     amount = realizedPnl（盈利为正，亏损为负）
     含完整 balance / frozen / margin_used 前后快照

  ⑦ 更新 t_position
     status       = CLOSED
     close_price  = markPrice
     close_reason = 1
     realized_pnl = realizedPnl
     closed_at    = now

  ⑧ 写 t_trade（trade_type=2）
     price        = markPrice
     quantity     = position.quantity
     realized_pnl = realizedPnl（与 t_position.realized_pnl 一致）
     fee          = 0

  ⑨ 回补 t_risk_exposure
     BUY 平仓：total_long_qty -= qty，net_exposure -= qty
     SELL 平仓：total_short_qty -= qty，net_exposure += qty

  ⑩ 写 t_outbox（trading.position.closed）

  ⑪ 事务提交后：OpenPositionSnapshotStore.remove(symbol, positionId)
```

## 五、QuoteDrivenEngine 强平触发（Step 4）

### 5.1 当前状态

`QuoteDrivenEngine` 当前只做两件事：

1. 保存最新价格快照
2. 读取 `OpenPositionSnapshotStore` 中的 OPEN 仓位数量

没有强平 / TP/SL / in-flight 去重。

### 5.2 改造后的触发逻辑

```text
processTick(payload):
  1. 更新价格快照
  2. 若 stale，只刷新快照不触发业务（沿用现有策略）
  3. 读取该 symbol 的所有 OPEN 持仓快照
  4. 过滤掉已处于 liquidation-in-flight 的 position
  5. 判断强平条件：
     - LONG：markPrice <= liquidationPrice
     - SHORT：markPrice >= liquidationPrice
  6. 命中的 positionId 进入 in-flight 注册表
  7. 按 userId 分组，提交到独立的异步执行器（不阻塞 Kafka listener 线程）
  8. 返回触发数量
```

### 5.3 并发策略

- 不做"按仓位无脑并行"（同用户多仓会争用同一 `t_account` 行）
- **按 `userId` 串行，跨用户并行**
- 实现：`@Async` + 按 `userId` 哈希的串行执行器，或应用层按 `userId` 分桶

### 5.4 异步执行与 Kafka offset 边界

- Kafka listener 线程"提交任务"后立即返回 ACK
- 异步执行失败由 `LiquidationExecutionService` 自管重试
- **不**把执行异常抛回 listener，避免触发 offset 回滚导致重复消费

### 5.5 in-flight 注册表

```java
interface LiquidationInFlightRegistry {
    boolean tryMark(long positionId);
    void clear(long positionId);
}
```

- 触发阶段：`tryMark` 成功的 position 才能进入执行
- 执行成功后：`OpenPositionSnapshotStore.remove` + `clear`
- 执行失败后：只 `clear`，仓位保留在快照中等待下轮 tick

### 5.6 重启恢复策略

进程内注册表重启丢失。启动时：

1. 从 MySQL 装载所有 `status=OPEN` 持仓到 `OpenPositionSnapshotStore`
2. 对每个仓位立即读取 Redis 最新价，若已满足强平条件，直接进入执行队列（而非等待下一个 tick）
3. 若 Redis 无最新价、最新价 `stale=true`，或价格时间戳按全局 `5s` 规则已判定为 `STALE`，则：
   - 不做立即强平
   - 记录 `WARN` 日志并保留仓位快照
   - 等待下一条新鲜 tick 到达后再参与强平判断
4. 不引入 `liquidation_pending_at` 持久标记，保持 schema 简洁

## 六、LiquidationExecutionService（Step 5）

### 6.1 与手动平仓的差异

| 维度 | 手动平仓 | 强平 |
|------|---------|------|
| 入口 | REST | QuoteDrivenEngine 异步任务 |
| 校验 | 不需要价格二次校验 | 必须二次校验 markPrice 仍满足强平条件 |
| 账务 | 同 §4.4 | 同 §4.4 + 负净值兜底 |
| `status` | CLOSED | LIQUIDATED |
| `close_reason` | 1 | 4 |
| `biz_type` | 8 | 9 |
| `trade_type` | 2 | 3 |
| 附加记录 | 无 | `t_liquidation_log` |

### 6.2 执行流程（同一 `@Transactional`）

```text
executeLiquidation(positionId, triggerMarkPrice):

  ① SELECT t_position WHERE id=? FOR UPDATE
     若 status != OPEN，幂等返回并清理 in-flight

  ② 按最新 margin / liquidationPrice 二次校验 triggerMarkPrice
     若已不满足强平条件（例如追加保证金后），取消执行并清理 in-flight

  ③ SELECT t_account WHERE user_id=? FOR UPDATE

  ④ 计算盈亏与兜底
     direction       = side==BUY ? +1 : -1
     realizedPnl     = (triggerMarkPrice - entryPrice) × quantity × direction
     balanceAfterRaw = account.balance + realizedPnl
     balanceAfter    = max(0, balanceAfterRaw)
     platformCov     = max(0, -balanceAfterRaw)

  ⑤ 更新 t_account
     balance     = balanceAfter
     margin_used = margin_used - position.margin
     frozen       不变
     version    += 1

  ⑥ 写 t_ledger（biz_type=9 liquidation_pnl）
     amount = realizedPnl（亏损为负，盈利为正）
     含完整前后快照

  ⑦ 更新 t_position
     status       = LIQUIDATED
     close_price  = triggerMarkPrice
     close_reason = 4
     realized_pnl = realizedPnl
     closed_at    = now

  ⑧ 写 t_trade（trade_type=3）
     price        = triggerMarkPrice
     quantity     = position.quantity
     realized_pnl = realizedPnl（与 t_position.realized_pnl 一致）
     fee          = 0

  ⑨ 回补 t_risk_exposure（与手动平仓一致）

  ⑩ 写 t_liquidation_log
     loss                  = max(0, -realizedPnl)
     fee                   = 0
     margin_released       = position.margin
     platform_covered_loss = platformCov

  ⑪ 写 t_outbox（trading.liquidation.executed）

  ⑫ 事务提交后：
     - OpenPositionSnapshotStore.remove(symbol, positionId)
     - LiquidationInFlightRegistry.clear(positionId)
```

### 6.3 字段语义澄清

- `t_liquidation_log.margin_released` = 本次账户侧释放的 `margin_used` 原值，即 `position.margin`。语义为"账户释放给用户的保证金"，不是"抵扣亏损后剩余保证金"。
- `t_ledger.biz_type=9` 命名为 `liquidation_pnl`，`amount = realizedPnl`，允许正负值，保证账本命名与金额语义一致。

### 6.4 负净值保护边界

| 场景 | balanceBefore | realizedPnl | balanceAfter | platformCoveredLoss |
|------|---|---|---|---|
| 轻度亏损 | 1000 | -50 | 950 | 0 |
| 盈亏归零 | 1000 | 0 | 1000 | 0 |
| 账户被打穿 | 20 | -50 | 0 | 30 |

### 6.5 强平手续费

一期 `fee = 0`，平台不额外收强平手续费。

## 七、追加保证金接口（Step 6）

### 7.1 REST 接口

```text
POST /api/v1/trading/positions/{positionId}/margin
Authorization: Bearer <accessToken>

Body: { "amount": "100.00" }
```

### 7.2 错误码

| 错误码 | 场景 |
|--------|------|
| `40001` | 可用余额不足 |
| `40004` | positionId 不存在或不属于当前用户 |
| `40007` | 持仓已关闭或已强平 |

> 说明：一期所有持仓均为 ISOLATED，**不**做 `margin_mode=ISOLATED` 的冗余校验；`40010` 专用于下单路径，不跨接口复用。

### 7.3 执行逻辑（同一 `@Transactional`）

```text
addIsolatedMargin(userId, positionId, amount):

  ① SELECT t_position WHERE id=? AND user_id=? FOR UPDATE
     校验 status=OPEN

  ② SELECT t_account WHERE user_id=? FOR UPDATE
     校验 available >= amount

  ③ 更新 t_account
     balance       不变
     frozen        不变
     margin_used  += amount
     version      += 1

  ④ 更新 t_position
     margin            += amount
     liquidation_price  = 重算

  ⑤ 写 t_ledger（biz_type=10）
     amount = +amount
     含完整快照

  ⑥ 事务提交后：OpenPositionSnapshotStore.upsert(position)
```

### 7.4 强平价重算公式

`LiquidationPriceCalculator` 扩展为通用重载：

```java
calculate(side, entryPrice, quantity, margin, maintenanceMarginRate)  // 通用
calculate(side, entryPrice, leverage, maintenanceMarginRate)          // 兼容旧调用
```

逐仓通用公式：

```text
BUY  liquidationPrice = entryPrice - margin/quantity + entryPrice × mmRate
SELL liquidationPrice = entryPrice + margin/quantity - entryPrice × mmRate
```

### 7.5 事件与推送

- 一期**不发** Outbox 事件
- 前端展示最新强平价走主动拉取持仓（WebSocket 推送不在本方案范围）

### 7.6 减少保证金

一期**不支持**，避免触发价格走位与强平条件的并发语义问题。

## 九、改造风险与注意事项

### 9.1 强平触发并发
- `processTick` 可能多线程并发
- 必须有 `LiquidationInFlightRegistry`
- 不能靠"直接从快照移除"当唯一去重手段

### 9.2 强平与追加保证金竞态
- 用户追加保证金与引擎强平可能同时发生
- 强平执行必须二次校验最新 `margin / liquidation_price`
- 若不再满足强平条件，放弃执行并清理 in-flight

### 9.3 账户更新并发
- `t_account` 一用户一账户
- 同用户多仓位强平争用同一账户行
- 执行器按 `userId` 串行 + `SELECT FOR UPDATE`
- 本方案统一使用悲观锁，不混用乐观锁（避免频繁冲突重试）

### 9.4 重启恢复
- in-flight 注册表为进程内状态，重启丢失
- 启动时全量装载 OPEN 持仓，并立即按当前 mark 过一遍强平条件

### 9.5 Kafka offset 与异步执行
- Listener 提交任务后立即 ACK
- 强平执行异常**不**抛回 listener
- 失败重试由 `LiquidationExecutionService` 自管

## 十二、最终判断

本方案在 V2 修订后可落地，核心前提：

1. **V4 Schema 一次性合并** `margin_mode` + 平仓字段（修复真实 V1/V2/V3 仍缺失的字段）
2. **手动平仓先于强平执行**，形成可复用的账务模板
3. **`t_account` 更新统一用 `SELECT FOR UPDATE`**，不混用乐观锁
4. **按 `userId` 串行，跨用户并行**
5. **in-flight 注册表 + 重启恢复**策略闭环
6. **文档与代码同步**：V4 落地后同步数据库蓝图、REST 规范、统一接口文档和状态机规范

满足这些前提后，逐仓模式可作为 `trading-core-service` 的受控增强，不会破坏 Stage 5 已完成的基础设施和账务语义。
