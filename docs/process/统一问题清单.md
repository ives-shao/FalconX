# FalconX 统一问题清单

## 1. 目的

本文件用于统一收敛 FalconX 仓库后续发现的问题、审计项、补充整改项和待修复事项。

后续若需要“读取问题并修复”，默认优先读取本文件，而不是再分散创建新的临时问题文档。

适用范围：

- 架构问题
- 文档不一致问题
- 数据库设计问题
- 接口契约问题
- 事件与幂等问题
- 代码实现问题
- 测试缺口问题

## 2. 使用规则

1. 新问题优先追加到本文件，不再分散到其他临时问题文档。
2. 每个问题都必须注明：
   - 优先级
   - 当前状态
   - 问题描述
   - 涉及范围
   - 期望修复方式
3. 修复时必须先判断问题是否真实存在，再更新状态。
4. 问题关闭后不删除，改为标记为 `已完成`，保留审计记录。

## 3. 状态约定

- `待确认`：问题已提出，但尚未核对是否真实存在
- `待修复`：问题已确认存在，等待实施
- `修复中`：问题正在处理
- `已完成`：问题已修复并验证通过
- `不成立`：问题经核对后不成立，不需要修复
- `后续阶段处理`：问题成立，但按计划留到后续阶段实现

## 4. 优先级约定

- `P0`：必须立即处理，不修会阻塞当前阶段
- `P1`：应尽快处理，会影响下一阶段或核心质量
- `P2`：优化项或完整性补齐项

  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/WalletDepositTransactionRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/MybatisWalletDepositTransactionRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/mapper/WalletDepositTransactionMapper.java`
  - `falconx-wallet-service/src/main/resources/mapper/wallet/WalletDepositTransactionMapper.xml`
  - `falconx-wallet-service/src/test/java/com/falconx/wallet/listener/Web3jChainDepositListenerTests.java`

### 期望修复

让 EVM listener 在确认窗口回扫时，同时加载该窗口内 owner 已落地的原始交易；
若已 `CONFIRMED` 的原生币入金在当前 canonical block 集合里消失，则生成 `reversed=true` 的 `ObservedDepositTransaction`，
交由应用层按既有状态机推进 `CONFIRMED -> REVERSED`。

### 核对结论

- 是否真实存在：`是`
- 结论说明：核对代码确认旧实现只处理“当前扫到的链上交易”，没有对已落地窗口交易做缺失对账，因此回滚识别入口缺失。

### 修复记录

- 修复内容：

  - `Web3jChainDepositListener` 现在会在确认窗口内同时加载 owner 已跟踪交易，并收集当前 canonical block 集合中的交易哈希
  - 对已 `CONFIRMED` 且在当前窗口彻底消失的原生币入金，会生成 `reversed=true` 的观察结果交给应用层
  - listener 仍然只处理原生币；`ERC20` 解析与更高阶确认推进未在本次范围内展开
- 验证结果：
  - `mvn -pl falconx-wallet-service -am test`
  - `mvn clean compile`

## [FX-050] wallet-service EVM 扫块按交易逐条查 owner 地址，形成 N+1 查询压力

- 优先级：`P1`
- 当前状态：`已完成`

这会把 listener 放大成“每条交易一次 owner 库查询”的 N+1 模式，和后续生产态性能要求不匹配。

  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/WalletAddressRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/MybatisWalletAddressRepository.java`
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/repository/mapper/WalletAddressMapper.java`
  - `falconx-wallet-service/src/main/resources/mapper/wallet/WalletAddressMapper.xml`
  - `falconx-wallet-service/src/test/java/com/falconx/wallet/listener/Web3jChainDepositListenerTests.java`

### 期望修复

让 EVM 监听器在每轮扫描开始时一次性批量加载当前链的启用平台地址快照，

### 核对结论

- 是否真实存在：`是`
- 结论说明：核对监听器实现确认此前按交易逐条调用地址 Repository，存在明显的 N+1 查询路径。

### 修复记录

- 修复内容：
  - `WalletAddressRepository` 新增按链批量加载启用地址集合的方法
  - MyBatis XML 已补齐对应批量查询 SQL
  - `Web3jChainDepositListener` 每轮同步时先预载地址快照，再按交易做内存匹配
  - 地址匹配统一做小写归一化，降低 EVM checksum 大小写差异带来的误判风险
- 验证结果：
  - `mvn -pl falconx-wallet-service -am test`
  - `mvn clean compile`

## [FX-011] `t_risk_exposure` UPSERT `net_exposure` double-count 计算 bug

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

`TradingRiskExposureMapper.xml` 中 `applyLongDelta` 和 `applyShortDelta` 的 `ON DUPLICATE KEY UPDATE` 语句存在财务计算错误。

MySQL `ON DUPLICATE KEY UPDATE` 中，同一 `UPDATE` 子句内，**后续列引用前面已更新过的列时得到的是更新后的值**。当前写法导致 `net_exposure` 每次被多加（或多减）一个 delta：

```sql
-- applyLongDelta（有问题）
total_long_qty = total_long_qty + VALUES(total_long_qty),          -- ① total_long_qty 已变为 old+delta
net_exposure   = (total_long_qty + VALUES(total_long_qty)) - total_short_qty  -- ② 此处 total_long_qty=old+delta, VALUES=delta
-- 实际结果：net_exposure = old_long + 2×delta - old_short  ← 多加了一个 delta
```

同理 `applyShortDelta` 也有对称的 double-count 问题。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：`falconx-trading-core-service/src/main/resources/mapper/trading/TradingRiskExposureMapper.xml`（第 33、53 行）

### 期望修复

`ON DUPLICATE KEY UPDATE` 内 `net_exposure` 直接引用**已更新后**的方向字段即可，不再重复加 `VALUES()`：

```sql
-- applyLongDelta
total_long_qty = total_long_qty + VALUES(total_long_qty),
net_exposure   = total_long_qty - total_short_qty,   -- total_long_qty 已是更新后的值

-- applyShortDelta
total_short_qty = total_short_qty + VALUES(total_short_qty),
net_exposure    = total_long_qty - total_short_qty,  -- total_short_qty 已是更新后的值
```

### 核对结论

- 是否真实存在：`是`
- 结论说明：直接读取 `TradingRiskExposureMapper.xml` 第 32-34 行和第 51-53 行，与 MySQL `ON DUPLICATE KEY UPDATE` 列求值语义对照，确认每次开仓后 `net_exposure` 偏差为 `+delta`（多头方向）或 `-delta`（空头方向）。

### 修复记录

- 修复内容：
  - `TradingRiskExposureMapper.xml` 的多头/空头 UPSERT 已改为基于更新后的 `total_long_qty / total_short_qty` 直接重算 `net_exposure`
  - 去掉了重复叠加 `VALUES(...)` 导致的净敞口偏移
- 验证结果：

## [FX-052] wallet 与 trading 仍按 `(chain, txHash)` 幂等，无法安全承载多 transfer 同交易

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

当前 `wallet-service` 的原始链事实和 `trading-core-service` 的业务入账事实都以 `(chain, txHash)` 作为唯一业务主键。  
这对原生币转账尚可工作，但对 ERC20 / SPL / TRC20 等“单笔链上交易包含多条 transfer”场景不成立，会导致：

- wallet owner 无法把同一 `txHash` 下的多条 transfer 稳定拆分为多条原始事实
- trading-core 仍会把多条 transfer 坍缩为同一个业务入账事实
- confirmed / reversed 幂等会出现覆盖、误判重复或漏入账

### 涉及范围

- 模块：`falconx-wallet-contract`
- 模块：`falconx-wallet-service`
- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-wallet-service/src/main/resources/db/migration/V1__init_wallet_schema.sql`
  - `falconx-trading-core-service/src/main/resources/db/migration/V1__init_trading_schema.sql`
  - `docs/database/falconx一期数据库设计.md`
  - `docs/event/Kafka事件规范.md`

### 期望修复

1. `t_wallet_deposit_tx` 增加：
   - `token_contract_address`
   - `log_index`
2. wallet owner 原始事实唯一键改为：
   - `(chain, tx_hash, log_index)`
3. wallet -> trading 的跨服务唯一业务主键统一改为：
   - `walletTxId`
4. `wallet.deposit.detected / confirmed / reversed` 事件 payload 增加：
   - `walletTxId`
5. `trading-core-service` 的 `t_deposit` 和业务入账 / 回滚幂等统一切换为：
   - `wallet_tx_id`

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取当前 `t_wallet_deposit_tx` 与 `t_deposit` schema、wallet 事件 payload 和 `TradingDepositCreditApplicationService`，确认两侧仍按 `(chain, txHash)` 做唯一性与幂等，无法承载多 transfer 同交易。

### 修复记录

- 修复内容：
  - 已在 `wallet-service` 增加 `token_contract_address`、`log_index`，并把原始链事实唯一键升级为 `(chain, tx_hash, log_index)`
  - 已在 `wallet.deposit.detected / confirmed / reversed` 事件 payload 中增加 `walletTxId`
  - 已在 `trading-core-service` 将 `t_deposit` 与业务入账 / 回滚幂等统一切换为 `wallet_tx_id`
  - 已同步数据库设计、事务与幂等规范、开发启动手册、当前开发计划与服务 README
- 验证结果：
  - `mvn -pl falconx-wallet-contract,falconx-wallet-service,falconx-trading-core-service -am test -Dsurefire.failIfNoSpecifiedTests=false`：通过
  - `mvn test`：待本轮统一回归
  - `mvn clean compile`：待本轮统一回归
