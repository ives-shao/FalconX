# FalconX 统一问题清单

## 1. 目的

本文件用于统一收敛 FalconX 当前仍需关注的活跃问题、审计项、补充整改项和待修复事项。

后续若需要“读取问题并修复”，默认先读本文件。若需要追溯已完成或不成立的问题，再读 [已归档问题清单](./archive/统一问题清单-已归档.md)。

适用范围：

- 架构问题
- 文档不一致问题
- 数据库设计问题
- 接口契约问题
- 事件与幂等问题
- 代码实现问题
- 测试缺口问题

## 2. 使用规则

1. 新问题优先追加到本文件，不再分散到其他临时问题文档。
2. 每个问题都必须注明：
   - 优先级
   - 当前状态
   - 问题描述
   - 涉及范围
   - 期望修复方式
3. 修复时必须先判断问题是否真实存在，再更新状态。
4. 问题状态进入 `已完成` 或 `不成立` 后，应移入 [已归档问题清单](./archive/统一问题清单-已归档.md)，保留审计记录。
5. `待确认 / 待修复 / 修复中 / 后续阶段处理` 问题继续保留在本文件，作为默认执行入口。

## 3. 状态约定

### 3.1 本文件默认保留的活跃状态

- `待确认`：问题已提出，但尚未核对是否真实存在
- `待修复`：问题已确认存在，等待实施
- `修复中`：问题正在处理
- `后续阶段处理`：问题成立，但按计划留到后续阶段实现

### 3.2 归档状态

- `已完成`：问题已修复并验证通过
- `不成立`：问题经核对后不成立，不需要修复

## 4. 优先级约定

- `P0`：必须立即处理，不修会阻塞当前阶段
- `P1`：应尽快处理，会影响下一阶段或核心质量
- `P2`：优化项或完整性补齐项

## [FX-041] PATCH 修改 TP/SL 与自动触发并发交叉时，旧快照条件可能在锁后失效却仍被执行

- 优先级：`P0`
- 当前状态：`已修复`

### 问题描述

自动平仓 / 强平链路最初按 `OpenPositionSnapshotStore` 中的旧快照先判定 `closeReason`，
随后在 `closePositionByTrigger(...)` 中锁定最新 DB 持仓直接落业务写路径。

这会留下一个真实竞态：
- tick 已按旧 snapshot 判定命中 `TP / SL / LIQUIDATION`
- 用户随后通过 `PATCH /api/v1/trading/positions/{positionId}` 修改或清空 TP/SL
- owner 写路径若不再基于最新 DB 持仓重新复核，就会把已经失效的旧触发条件错误执行成真实平仓

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingPositionCloseApplicationService.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/controller/TradingControllerIntegrationTests.java`

### 期望修复

- `closePositionByTrigger(...)` 锁到最新 `OPEN` 持仓后，必须基于“当前 DB 持仓 + 当前 quote.mark”重新判定有效触发结果
- 若已不再满足触发条件，必须直接 `skip`，不写任何 owner 表
- 若重新判定出的原因与入参不同，必须以重新判定结果为准并记录日志

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮代码核对确认，旧自动触发写路径只使用前置判定出的 `closeReason`，锁到最新 DB 持仓后未再次复核当前 TP/SL，真实存在按旧快照误平仓风险。

### 修复记录

- 修复内容：
  - `TradingPositionCloseApplicationService.closePositionByTrigger(...)` 已在锁定最新 `OPEN` 持仓后重新运行触发规则评估
  - 若当前 DB 持仓已不再满足触发条件，则直接返回 `null`，不写 `t_position / t_trade / t_ledger / t_outbox / t_risk_exposure`
  - 若复核结果与入参 `closeReason` 不一致，则以复核结果为准继续执行业务写路径，并记录日志

### 验证结果

- `TradingControllerIntegrationTests.shouldSkipTriggeredCloseWhenPatchClearsTakeProfitBeforeOwnerWritePathRevalidation` 已验证：
  - 用户先通过真实 `PATCH /api/v1/trading/positions/{positionId}` 清空 `takeProfitPrice`
  - 之后即使触发路径仍携带旧的 `TAKE_PROFIT` 原因进入 owner 写路径，也会被最新 DB 持仓复核拦下
  - 持仓保持 `OPEN`，`t_trade / t_ledger / t_outbox` 不新增，`OpenPositionSnapshotStore` 也已同步成清空后的最新值

## [FX-042] QuoteDrivenEngine 在同一 tick 下处理多持仓时，单笔失败会中断后续触发并丢失部分结果

- 优先级：`P0`
- 当前状态：`已修复`

### 问题描述

旧 `QuoteDrivenEngine` 在遍历同一 `symbol` 下的 `OPEN` 持仓时，任意一笔
`closePositionByTrigger(...)` 抛错都会直接中断整个循环。

这会带来两个问题：
- 同一 tick 下后续本应命中的其他持仓不会继续处理
- 已成功触发的持仓结果与首个失败信息无法同时保留，调用方只能拿到“中途失败”而看不到剩余持仓是否已被继续处理

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/engine/QuoteDrivenEngine.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/TradingPersistenceIntegrationTests.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/TradingLiquidationIntegrationTests.java`

### 期望修复

- 同一 tick 下每个持仓必须独立执行触发关闭
- 单笔失败不能阻断同 tick 其他持仓的处理
- 引擎必须记录第一条失败并继续处理剩余持仓
- 循环结束后若存在失败，再把第一条失败向上抛出；已成功提交的持仓结果必须保留

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮代码核对确认，旧实现对多持仓逐一关闭时没有做失败隔离，任意一笔异常都会提前终止本轮 tick 的剩余处理。

### 修复记录

- 修复内容：
  - `QuoteDrivenEngine` 已改为按 position 独立执行关闭逻辑
  - 单笔失败时记录首个异常并继续处理剩余持仓
  - 本轮 tick 处理完成后，若存在失败，再向上抛出首个异常
  - 强平失败整笔回滚口径继续保留在 `TradingLiquidationIntegrationTests`，作为唯一正式回滚语义

### 验证结果

- `TradingPersistenceIntegrationTests.shouldContinueProcessingOtherTriggeredPositionsWhenOneTriggeredCloseFails` 已验证：
  - 同一 `symbol` 下两笔 `OPEN` 持仓同时命中时，一笔因 `t_risk_exposure` 更新失败整笔回滚
  - 另一笔仍成功完成触发关闭并保留业务结果
  - 本轮循环结束后仍会把第一条失败向上抛出，不会静默吞掉异常
- `TradingLiquidationIntegrationTests.shouldRollbackLiquidationWhenRiskExposureUpdateFails` 已继续验证强平失败的唯一正式回滚口径：
  - `t_position` 保持 `OPEN`
  - `trade_type=3 / biz_type=9 / liquidation_log / trading.liquidation.executed` 均为 `0`
  - `t_account.balance / margin_used` 回到原值
  - `OpenPositionSnapshotStore` 中原快照仍存在

## [FX-043] MarketPriceTickEventConsumer 不能在 Kafka listener 回调线程中直接执行业务写路径

- 优先级：`P0`
- 当前状态：`已修复`

### 问题描述

`falconx.market.price.tick` 属于高频事件。若在 Kafka listener 回调线程中直接进入
`QuoteDrivenEngine -> owner DB 写路径`，会把高频触发判定、TP/SL / 强平结算、`t_outbox`
写入和风险暴露更新都压在 listener 线程上执行。

这会带来两个直接风险：
- 违反回调线程不得直接执行业务写链路的正式约束
- 线程上下文（尤其是 `traceId`）与日志连续性容易在跨线程实现中丢失

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/consumer/MarketPriceTickEventConsumer.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/consumer/MarketPriceTickEventConsumerTests.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/consumer/TradingKafkaEventListenerTests.java`

### 期望修复

- `MarketPriceTickEventConsumer` 必须先切到 trading-core 自管执行器，再进入 `QuoteDrivenEngine`
- 触发后的 owner 写失败必须继续向上抛出，不允许只记日志后吞掉
- `traceId` 必须在执行线程中保持可读，保证日志连续性

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮核对确认，`falconx.market.price.tick` 触发的是 owner 级业务写链路，必须从 Kafka listener 线程切到 trading-core 自管执行器后再执行；该约束现已由实现和测试共同固定。

### 修复记录

- 修复内容：
  - `MarketPriceTickEventConsumer` 已切换为先提交到 trading-core 自管执行器，再进入 `QuoteDrivenEngine`
  - 当前实现保留异常上抛语义，不把 TP/SL / 强平 owner 写失败降级成“只记日志后继续”
  - 已补充线程切换与 `traceId` 保持测试，以及真实 listener 到 consumer 的委托测试

### 验证结果

- `MarketPriceTickEventConsumerTests.shouldSwitchToManagedExecutorThreadAndKeepTraceIdAcrossExecution` 已验证：
  - `QuoteDrivenEngine` 不在 listener 调用线程执行
  - 执行线程名以 `trading-price-tick-` 开头
  - `traceId` 在线程切换后仍可被下游读取
- `TradingKafkaEventListenerTests.shouldDelegateMarketPriceTick` 已验证：
  - Kafka listener 真实入口会把 `falconx.market.price.tick` 委托给 `MarketPriceTickEventConsumer`
  - 当前结论只应表述为“已切到自管执行器线程”，不能误写成“已完全异步解耦 listener”

## [FX-023] 交易时间校验未区分"开仓"与"平仓/强平"，非交易时段用户无法手动平仓

- 优先级：`P0`
- 当前状态：`已修复`

### 问题描述

`DefaultTradingScheduleService.isTradable(symbol)` 返回单一布尔值，开仓与平仓共用同一校验结果。此前交易模块没有真实手动平仓入口，阶段风险在于一旦平仓或强平链路落地，仍会继续沿用开仓交易时间限制，导致在非交易时段：
- 手动平仓请求被错误阻塞
- 后续强平执行路径同样存在被交易时间校验误伤的风险

逐仓改造方案 §4.1 明确规定：非交易时段只拒绝 OPEN，平仓和强平应始终放行。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/impl/DefaultTradingScheduleService.java`
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingOrderPlacementApplicationService.java`
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingPositionCloseApplicationService.java`
  - `docs/domain/状态机规范.md`

### 期望修复

`TradingScheduleService` 拆分接口为两个方法：
- `isOpenAllowed(symbol)`：开仓校验，含交易时间、节假日等
- `isCloseAllowed(symbol)`：平仓/强平不受交易时间窗口阻塞

开仓链路继续按 `isOpenAllowed` 拒绝非交易时段；手动平仓链路改走独立 close 语义。

### 核对结论

- 是否真实存在：`是`
- 结论说明：历史实现只有统一 `isTradable` 语义，且 Stage 7 前没有真实手动平仓入口；若直接沿用旧设计实现平仓，会把开仓交易时间限制错误带入平仓路径。

### 修复记录

- 修复内容：
  - `TradingScheduleService` 已拆分为 `isOpenAllowed / isCloseAllowed`
  - `DefaultTradingRiskService` 的开仓风控改为只调用 `isOpenAllowed`
  - 已新增 `POST /api/v1/trading/positions/{positionId}/close` 真实手动平仓入口，非交易时段不再受开仓时间校验阻塞
  - 手动平仓在同事务内完成账户结算、`t_ledger.biz_type=8`、持仓终态写入、`t_trade.trade_type=2` 和 `t_risk_exposure` 回补
- 验证结果：
  - `TradingControllerIntegrationTests.shouldAllowManualCloseEvenWhenHolidayBlocksNewOpenOrders` 已验证：节假日全休场景下手动平仓返回成功，而不是 `40008`
  - `DefaultTradingScheduleServiceTests.shouldAllowManualCloseWhenHolidayFullCloseBlocksNewOpen` 已验证：同一时刻 `isOpenAllowed=false`，`isCloseAllowed=true`
  - `TradingControllerIntegrationTests.shouldClosePositionSuccessfullyAndRemoveSnapshot` 已验证：手动平仓成功后 `t_account / t_ledger / t_position / t_trade / t_risk_exposure` 与 `OpenPositionSnapshotStore` 同步收敛到终态

## [FX-024] Gateway 无请求限流，缺乏基础防刷与 DDoS 防护

- 优先级：`P0`
- 当前状态：`部分修复`

### 问题描述

`falconx-gateway` 的路由配置中没有任何限流过滤器（`RequestRateLimiter`、`RedisRateLimiter` 等），所有端点对外完全开放，无每 IP / 每用户的速率限制。攻击者可对登录接口暴力枚举密码、对行情接口发起大量查询，耗尽后端资源。

### 涉及范围

- 模块：`falconx-gateway`
- 文件：
  - `falconx-gateway/src/main/resources/application.yml`（路由配置）
  - `docs/security/安全规范.md`

### 期望修复

Stage 6B 统一处理，至少对以下端点添加限流：
- `/api/identity/**`（登录、注册）：每 IP 每分钟 20 次
- `/api/trading/**`（下单）：每用户每秒 10 次
- 全局兜底：每 IP 每分钟 200 次

推荐使用 Spring Cloud Gateway 的 `RedisRateLimiter`（复用已有 Redis 连接）。

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 `falconx-gateway/src/main/resources/application.yml` 确认路由配置中无任何 `RequestRateLimiter` 或类似限流过滤器。

### 修复记录

- 修复内容：
  - `falconx-gateway` 已新增 `GatewayRateLimitFilter`，对 `/api/v1/auth/login` 与 `/api/v1/auth/register` 落地每 IP 每分钟 20 次限流
  - 认证入口限流返回统一 JSON 响应，登录超限返回 `10003`，注册超限返回 `10004`
  - 当前轮未实现 `/api/v1/trading/**` 每用户每秒 10 次与全局每 IP 每分钟 200 次兜底限流
- 验证结果：
  - `GatewayRoutingIntegrationTests.shouldRateLimitAuthRequestByIpAtGateway` 已验证第 21 次登录请求返回 `429 / 10003`

## [FX-036] `wallet.deposit.reversed` 在 owner 入金事实不存在时被错误提前确认

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

`TradingDepositReversalApplicationService` 旧实现存在一个真实幂等缺陷：

一旦后续 `wallet.deposit.confirmed` 才到达，之前那条 `reversed` 可能已经被错误吞掉，导致业务入金事实无法被及时回滚。

  - `falconx-trading-core-service/src/test/java/com/falconx/trading/TradingKafkaWalletDepositIntegrationTests.java`

当 `wallet.deposit.reversed` 先到而 owner 入金事实不存在时：
- 不允许写成功 `t_inbox`

- 是否真实存在：`是`

### 修复记录

- 修复内容：
  - `TradingDepositReversalApplicationService` 已改为：当 `walletTxId` 对应交易域入金事实不存在时直接抛错，不再写成功 `t_inbox`

## [FX-040] 手动平仓成功响应把 `account.openPositions` 错误硬编码为空数组

- 优先级：`P1`
- 当前状态：`已修复`

### 问题描述

`TradingPositionController` 在手动平仓成功响应中直接把 `TradingAccountResponse.openPositions`
写成 `List.of()`，没有回显当前用户剩余的 `OPEN` 持仓视图。

这会破坏账户快照语义：
- 同一用户若仍有其他 `OPEN` 持仓，平仓成功响应会错误表现成“没有任何持仓”
- 北向接口的账户快照在 `GET /accounts/me` 与 `POST /positions/{positionId}/close` 之间出现不一致

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/controller/TradingPositionController.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/controller/TradingControllerIntegrationTests.java`

### 期望修复

- 手动平仓成功响应中的 `account.openPositions` 必须回显当前用户剩余的 `OPEN` 持仓视图
- 必须继续复用 owner 数据来源，不允许使用静态列表或把该字段偷换成“本次关闭的单持仓结果”

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮代码核对确认，旧控制器在 `toResponse(...)` 中把 `openPositions` 固定写成空数组，真实存在语义漂移。

### 修复记录

- 修复内容：
  - `TradingPositionController` 已改为复用 `OpenPositionSnapshotStore + TradingQuoteSnapshotRepository` 组装账户快照
  - 平仓成功响应中的 `account.openPositions` 现已回显当前用户剩余的 `OPEN` 持仓视图
- 验证结果：
  - `TradingControllerIntegrationTests.shouldReturnRemainingOpenPositionsInCloseSuccessResponse` 已验证：
    - 同一用户同时持有两笔 `OPEN` 持仓时，关闭其中一笔后，成功响应中的 `account.openPositions` 仍保留另一笔持仓
    - 被关闭持仓进入 `CLOSED`，另一笔持仓保持 `OPEN`
