# FalconX 统一问题清单

## 1. 目的

本文件用于统一收敛 FalconX 当前仍需关注的活跃问题、审计项、补充整改项和待修复事项。

后续若需要“读取问题并修复”，默认先读本文件。若需要追溯已完成或不成立的问题，再读 [已归档问题清单](./archive/统一问题清单-已归档.md)。

适用范围：

- 架构问题
- 文档不一致问题
- 数据库设计问题
- 接口契约问题
- 事件与幂等问题
- 代码实现问题
- 测试缺口问题

## 2. 使用规则

1. 新问题优先追加到本文件，不再分散到其他临时问题文档。
2. 每个问题都必须注明：
   - 优先级
   - 当前状态
   - 问题描述
   - 涉及范围
   - 期望修复方式
3. 修复时必须先判断问题是否真实存在，再更新状态。
4. 问题状态进入 `已完成` 或 `不成立` 后，应移入 [已归档问题清单](./archive/统一问题清单-已归档.md)，保留审计记录。
5. `待确认 / 待修复 / 修复中 / 后续阶段处理` 问题继续保留在本文件，作为默认执行入口。

## 3. 状态约定

### 3.1 本文件默认保留的活跃状态

- `待确认`：问题已提出，但尚未核对是否真实存在
- `待修复`：问题已确认存在，等待实施
- `修复中`：问题正在处理
- `后续阶段处理`：问题成立，但按计划留到后续阶段实现

### 3.2 归档状态

- `已完成`：问题已修复并验证通过
- `不成立`：问题经核对后不成立，不需要修复

## 4. 优先级约定

- `P0`：必须立即处理，不修会阻塞当前阶段
- `P1`：应尽快处理，会影响下一阶段或核心质量
- `P2`：优化项或完整性补齐项

## [FX-028] 注册流程无邮箱验证，账号身份可信度低

- 优先级：`P1`
- 当前状态：`后续阶段处理`

### 问题描述

`IdentityRegistrationApplicationService.register()` 保存用户后直接设置状态为 `PENDING_DEPOSIT`，未发送邮箱验证邮件，不要求用户点击链接确认邮箱真实性。攻击者可用虚假邮箱批量注册账号；CFD 平台合规层面通常需要验证用户联系方式。

### 涉及范围

- 模块：`falconx-identity-service`
- 文件：
  - `falconx-identity-service/src/main/java/com/falconx/identity/application/IdentityRegistrationApplicationService.java`
  - `docs/domain/状态机规范.md`（用户状态机）

### 期望修复

Stage 6B 处理：
1. 注册后状态改为 `PENDING_EMAIL_VERIFY`
2. 发送包含有时效 token 的验证邮件
3. 用户点击后状态流转为 `PENDING_DEPOSIT`
4. 状态机规范文档同步更新

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 `IdentityRegistrationApplicationService` 确认注册流程无邮件发送逻辑，用户注册后直接进入 `PENDING_DEPOSIT` 状态。

## [FX-031] `t_risk_config` 中 `max_position_per_user` 与 `max_position_total` 从未在风控路径中检查

- 优先级：`P1`
- 当前状态：`后续阶段处理`

### 问题描述

数据库 `t_risk_config` 表已定义 `max_position_per_user`（单用户最大持仓数）和 `max_position_total`（全平台最大持仓数）字段，但 `DefaultTradingRiskService.validateOrderRequest()` 完全没有读取和检查这两个字段。

实际风控路径只校验：symbol 有效性、交易时间、报价新鲜度、数量/杠杆有效性、可用保证金是否足够。

作为 B-book 平台，持仓数量上限是控制风险敞口规模的基础手段，未被使用相当于该风控参数形同虚设。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/impl/DefaultTradingRiskService.java`（`validateOrderRequest` 方法）
  - `falconx-trading-core-service/src/main/resources/mapper/trading/TradingRiskConfigMapper.xml`

### 期望修复

`DefaultTradingRiskService` 在通过保证金校验后，补充以下检查：
1. 查询当前用户 OPEN 持仓数，若 `>= max_position_per_user` 则拒绝（错误码 `30005 POSITION_LIMIT_REACHED`）
2. 查询全平台 OPEN 持仓总数，若 `>= max_position_total` 则拒绝（错误码 `30006 PLATFORM_POSITION_LIMIT_REACHED`）
3. 补充两个错误码到 `CommonErrorCode`

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 `DefaultTradingRiskService.validateOrderRequest` 方法，确认没有查询 `t_risk_config.max_position_per_user` 或 `max_position_total` 的任何逻辑。

## [FX-035] 钱包地址重复分配给同一用户后无隐私隔离，链上历史可关联

- 优先级：`P2`
- 当前状态：`后续阶段处理`

### 问题描述

当前 `SequentialWalletAddressAllocationService` 对每个 `(user_id, chain)` 仅分配一个地址（`uk_user_chain` 唯一约束）。用户所有入金都使用同一链上地址，链上分析工具可轻松关联该用户的全部资金来源与历史交易。对于 CFD 平台，用户资金链上隐私属于合规敏感项，尤其是 ERC-20 USDT 的 Ethereum 链。

### 涉及范围

- 模块：`falconx-wallet-service`
- 文件：
  - `falconx-wallet-service/src/main/java/com/falconx/wallet/service/impl/SequentialWalletAddressAllocationService.java`
  - `falconx-wallet-service/src/main/resources/db/migration/V1__init_schema.sql`（`t_wallet_address` 表结构）

### 期望修复

后续阶段规划：
- 放开 `uk_user_chain` 唯一约束，允许每用户每链拥有多个地址
- 每次新入金请求分配新的未使用地址（HD 钱包派生路径递增）
- 旧地址仍监控，新入金引导至新地址

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 `t_wallet_address` 表结构确认 `uk_user_chain` 唯一约束；服务层逻辑确认只维护单地址。
