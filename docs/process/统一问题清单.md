# FalconX 统一问题清单

## 1. 目的

本文件用于统一收敛 FalconX 当前仍需关注的活跃问题、审计项、补充整改项和待修复事项。

后续若需要“读取问题并修复”，默认先读本文件。若需要追溯已完成或不成立的问题，再读 [已归档问题清单](./archive/统一问题清单-已归档.md)。

适用范围：

- 架构问题
- 文档不一致问题
- 数据库设计问题
- 接口契约问题
- 事件与幂等问题
- 代码实现问题
- 测试缺口问题

## 2. 使用规则

1. 新问题优先追加到本文件，不再分散到其他临时问题文档。
2. 每个问题都必须注明：
   - 优先级
   - 当前状态
   - 问题描述
   - 涉及范围
   - 期望修复方式
3. 修复时必须先判断问题是否真实存在，再更新状态。
4. 问题状态进入 `已完成` 或 `不成立` 后，应移入 [已归档问题清单](./archive/统一问题清单-已归档.md)，保留审计记录。
5. `待确认 / 待修复 / 修复中 / 后续阶段处理` 问题继续保留在本文件，作为默认执行入口。

## 3. 状态约定

### 3.1 本文件默认保留的活跃状态

- `待确认`：问题已提出，但尚未核对是否真实存在
- `待修复`：问题已确认存在，等待实施
- `修复中`：问题正在处理
- `后续阶段处理`：问题成立，但按计划留到后续阶段实现

### 3.2 归档状态

- `已完成`：问题已修复并验证通过
- `不成立`：问题经核对后不成立，不需要修复

## 4. 优先级约定

- `P0`：必须立即处理，不修会阻塞当前阶段
- `P1`：应尽快处理，会影响下一阶段或核心质量
- `P2`：优化项或完整性补齐项

## [FX-023] 交易时间校验未区分"开仓"与"平仓/强平"，非交易时段用户无法手动平仓

- 优先级：`P0`
- 当前状态：`后续阶段处理`

### 问题描述

`DefaultTradingScheduleService.isTradable(symbol)` 返回单一布尔值，开仓与平仓共用同一校验结果。当前 `TradingOrderPlacementApplicationService` 对所有订单类型统一调用此方法，导致在非交易时段：
- 手动平仓请求被以 `40008 TRADING_TIME_RESTRICTED` 拒绝
- 强平引擎的执行路径同样受阻

逐仓改造方案 §4.1 明确规定：非交易时段只拒绝 OPEN，平仓和强平应始终放行。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/impl/DefaultTradingScheduleService.java`
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingOrderPlacementApplicationService.java`
  - `docs/domain/状态机规范.md`

### 期望修复

`TradingScheduleService` 拆分接口为两个方法：
- `isOpenAllowed(symbol)`：开仓校验，含交易时间、节假日等
- `isCloseAlwaysAllowed(symbol)`：平仓/强平始终返回 `true`（或仅做品种存在性校验）

`PlacementApplicationService` 按订单方向选用对应方法。

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 `DefaultTradingScheduleService` 确认 `isTradable` 无 OPEN/CLOSE 区分；`PlacementApplicationService` 对所有订单统一调用该方法。

## [FX-026] B-book 对冲决策信息严重缺失，平台风险敞口无法实时感知

- 优先级：`P0`
- 当前状态：`后续阶段处理`

### 问题描述

FalconX 作为 B-book 平台，净敞口超过阈值时需要向外部对冲（A-book）。当前系统：

1. `t_risk_exposure` 仅有 `total_long_qty / total_short_qty / net_exposure`，无按 `symbol + notional_value` 的美元敞口换算
2. 无对冲阈值配置表（应存入 `t_risk_config` 或独立表）
3. 无对冲触发日志表（对冲请求/成交记录不可追溯）
4. 无报警机制（净敞口超阈值时无任何通知）

在一期 1000 用户规模下，所有用户方向一致时净敞口可快速累积，平台无感知窗口。

### 涉及范围

- 模块：`falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/resources/db/migration/`（V1–V3 中无对冲相关表）
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/service/impl/DefaultTradingRiskService.java`
  - `docs/database/falconx一期数据库设计.md`

### 期望修复

Stage 7 前补充以下基础支撑：
1. `t_risk_config` 新增 `hedge_threshold_usd` 字段（或独立 `t_hedge_config` 表）
2. `t_risk_exposure` 增加 `net_exposure_usd`（按 mark_price 实时换算）
3. 新增 `t_hedge_log` 表记录对冲动作
4. `DefaultTradingRiskService` 在开仓后计算 `net_exposure_usd`，超过阈值时触发告警日志（先接日志，后接真实对冲接口）

### 核对结论

- 是否真实存在：`是`
- 结论说明：读取 V1–V3 Flyway 脚本，确认无对冲阈值、对冲日志、对冲触发相关表；`DefaultTradingRiskService` 无任何净敞口美元换算逻辑。

## [FX-036] `wallet.deposit.reversed` 在 owner 入金事实不存在时被错误提前确认

- 优先级：`P0`
- 当前状态：`已完成`

### 问题描述

`TradingDepositReversalApplicationService` 旧实现存在一个真实幂等缺陷：
当 `wallet.deposit.reversed` 先到、而 `walletTxId` 对应的交易域入金事实尚不存在时，
代码会先把该 `eventId` 写入成功 `t_inbox`，随后仅记录告警并返回。

这会违反《事务与幂等规范》中“owner 业务对象不存在时必须消费失败并重试，不能提前标记成功”的要求。
一旦后续 `wallet.deposit.confirmed` 才到达，之前那条 `reversed` 可能已经被错误吞掉，导致业务入金事实无法被及时回滚。

### 涉及范围

- 文档：
  - `docs/architecture/事务与幂等规范.md`
- 模块：
  - `falconx-trading-core-service`
- 文件：
  - `falconx-trading-core-service/src/main/java/com/falconx/trading/application/TradingDepositReversalApplicationService.java`
  - `falconx-trading-core-service/src/test/java/com/falconx/trading/TradingKafkaWalletDepositIntegrationTests.java`

### 期望修复

当 `wallet.deposit.reversed` 先到而 owner 入金事实不存在时：
- 不允许写成功 `t_inbox`
- 必须抛错，让 Kafka 保持失败并走重试
- 待 `wallet.deposit.confirmed` 完成入账后，之前失败的 `reversed` 应可在重试中成功完成业务回滚

### 核对结论

- 是否真实存在：`是`
- 结论说明：本轮通过代码核对与真实 Kafka 入口集成测试确认，该问题真实存在；其触发条件正是 `reversed` 先到、`confirmed` 后到的乱序场景。

### 修复记录

- 修复内容：
  - `TradingDepositReversalApplicationService` 已改为：当 `walletTxId` 对应交易域入金事实不存在时直接抛错，不再写成功 `t_inbox`
  - 新增真实 Kafka 入口集成测试，已验证：
    - `wallet.deposit.confirmed` 真消费并完成业务入账
    - duplicate confirmed 不重复入账
    - `wallet.deposit.reversed` 真消费并完成业务回滚
    - `reversed` 先到时初始阶段不写成功 `t_inbox`；待 `confirmed` 到达后，失败的 `reversed` 会在重试中成功完成 reversal
- 验证结果：
  - `wallet.deposit.confirmed` 进入真实 Kafka 入口后，会由 `TradingKafkaEventListener` 真消费，并写入 `t_inbox / t_deposit / t_outbox / t_account`，同时产出 `trading.deposit.credited` Kafka 消息。
