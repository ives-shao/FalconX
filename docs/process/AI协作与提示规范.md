# FalconX AI 协作与提示规范

## 1. 目的

本文件用于统一 FalconX 仓库内 AI 与人工协作时的**文档优先级、真源归属、冲突处理、最小阅读集、默认确认边界和验证深度**。

本文件解决的核心问题不是“再列一遍扁平优先级”，而是把不同类型的信息拆开治理：

- **正式契约**不能被 Prompt、README、计划文档改写。
- **当前阶段状态**不能由单个服务 README 或场景 Prompt 拍板。
- **冲突出现时**必须先识别“当前陈述属于哪一类信息”，再回到该类信息的真源文档，而不是按“哪个文档写得更新、写得更长、看起来更具体”来判断。

本文件是 `AGENTS.md` 下的默认执行策略文档，补充“怎么读、怎么判、怎么落地”，**不替代**仓库正式规范。

---

## 2. 适用范围

适用于：

- AI 编码代理（包括 Codex、Claude 等）
- 人工维护者
- 文档修订、问题审计、代码实现、测试验证、阶段收口、Prompt 编写等全部协作场景

---

## 3. 文档分层与职责

FalconX 不再使用“一个扁平优先级表处理所有问题”的方式。不同文档负责不同类型的信息。

### 3.1 执行硬约束

以下文件负责仓库级执行红线：

- `AGENTS.md`
- `SKILLS.md`
- `docs/process/完成定义.md`

它们回答的是：

- 开始任务前必须先读什么
- 哪些改动必须走对应 Skill
- 什么情况下任务才允许标记完成
- 是否需要测试、日志、回滚点、文档同步

### 3.2 正式规范

以下文件负责正式契约与正式边界：

- 架构与 owner 边界：
  - `docs/architecture/falconx一期网关-服务-数据库架构方案.md`
- 实施顺序、阶段定义、最小产出、最小验收基线、环境前置：
  - `docs/setup/开发启动手册.md`
- 数据库与 SQL 蓝图：
  - `docs/database/falconx一期数据库设计.md`
  - `docs/sql/V1__init_schema.sql`
  - `docs/sql/V2__seed_symbols.sql`
  - `docs/sql/CH_V1__market_analytics.sql`
- 接口、安全、事件与状态：
  - `docs/api/REST接口规范.md`
  - `docs/api/WebSocket接口规范.md`
  - `docs/api/FalconX统一接口文档.md`
  - `docs/security/安全规范.md`
  - `docs/event/Kafka事件规范.md`
  - `docs/domain/状态机规范.md`
  - `docs/architecture/事务与幂等规范.md`
  - `docs/market/tiingo报价接入契约.md`
- 编码、日志、测试与治理：
  - `docs/architecture/falconx编码与测试规范.md`
  - `docs/architecture/日志打印规范.md`
  - `docs/process/统一问题清单.md`
  - `docs/adr/README.md`

### 3.3 状态文档

以下文件负责**当前时点**的阶段状态、实施顺序、阻塞项和下一步：

- `docs/setup/当前开发计划.md`

它回答的是：

- 当前正式阶段结论是什么
- 哪些阶段已经收口，哪些还没收口
- 当前剩余阻塞项是什么
- 下一步先做什么

它**不负责**重新定义架构、owner、接口、数据库、事件、状态机等正式契约。

### 3.4 摘要 / 导航 / 场景提示文档

以下文件只允许承担“摘要、导航、场景执行提示”职责：

- 根 `README.md`
- 各服务 `README.md`
- `docs/design/codex-prompt.md`
- `docs/process/README.md`
- `docs/process/规则速查表.md`
- 其他场景 Prompt 文档

这些文档：

- 可以摘要
- 可以导航
- 可以说明本模块事实
- 可以给出执行模板

但**不能定义正式契约**，也**不能单独拍板当前阶段完成状态**。

## 4. 字段级真源矩阵（核心规则）

当文档互相冲突时，先判断“当前要判断的到底是什么信息”，再按下面的真源矩阵处理。

| 信息类型 | 唯一真源 / 第一真源 | 可引用的从属文档 | 冲突时怎么判 |
| --- | --- | --- | --- |
| 服务边界、owner 边界、调用方向、分层结构 | 架构方案 | README、Prompt、当前开发计划 | 一律以架构方案为准；下游文档视为过期摘要 |
| 阶段定义、实施顺序、最小产出、最小验收基线、环境前置条件 | 开发启动手册 | README、当前开发计划、Prompt | 一律以开发启动手册为准；状态文档只能说明“当前做到哪”，不能改定义 |
| 当前正式阶段、当前已完成/未完成、阻塞项、下一步顺序、当前分支基线 | 当前开发计划 | README、Prompt、阶段总结 | 一律以当前开发计划为准；README/Prompt 只能摘要当前状态，不能另立口径 |
| 数据库 owner、表结构、字段语义、索引、基线 SQL、migration 语义 | 数据库设计 + SQL 蓝图 + 运行时 migration | README、计划文档 | 以正式数据库文档和 SQL 为准；计划文档不得改写 schema |
| REST 协议、路径、方法、请求头、参数、响应结构、错误码 | REST 接口规范 | 统一接口文档、README、Prompt | 字段级冲突时 REST 规范优先；统一接口文档负责汇总，不得反向改写 REST 规范 |
| WebSocket 握手、鉴权、订阅协议、推送结构、连接限制 | WebSocket 接口规范 | 统一接口文档、README、Prompt | 字段级冲突时 WS 规范优先 |
| 接口总览、模块归属、上下游关系、业务说明 | 统一接口文档 | README、Prompt | 统一接口文档是汇总真源，但字段级协议仍服从 REST / WS 正式规范 |
| Topic、payload、分区键、生产/消费 owner、事件语义 | Kafka 事件规范 | README、当前开发计划、Prompt | 一律以 Kafka 事件规范为准；必要时再回到冻结 contract 与实现代码交叉核对 |
| 状态迁移、状态码语义、交易/订单/持仓状态规则 | 状态机规范 | README、计划文档、Prompt | 一律以状态机规范为准 |
| 幂等、Outbox / Inbox、事务边界、重试语义 | 事务与幂等规范 | README、计划文档、Prompt | 一律以事务与幂等规范为准 |
| 编码、测试、日志、完成判定 | 编码与测试规范 + 日志规范 + 完成定义 | README、Prompt | 一律以正式治理文档为准 |
| 模块职责摘要、运行方式、样例命令、本模块事实说明 | 各服务 README | Prompt、速查表 | 仅限摘要与导航；不得定义项目正式阶段或改写正式契约 |
| 场景执行方式、输出模板、协作节奏 | Prompt 文档 | README | 只能定义“怎么做”，不能定义“规则是什么” |

### 4.1 必须特别区分的两个概念

#### A. “阶段定义”与“当前阶段状态”不是一回事

- **阶段定义 / 最小验收基线**：看 `开发启动手册`
- **当前已经推进到哪一步 / 当前正式阶段结论**：看 `当前开发计划`

如果两者写法冲突，处理原则是：

- 判断“某个阶段应该满足什么条件”时，以 `开发启动手册` 为准。
- 判断“仓库当前到底处于哪个阶段、哪些事项本轮已完成”时，以 `当前开发计划` 为准。

#### B. “模块事实”与“项目结论”不是一回事

- 服务 `README` 可以描述模块当前已经存在的代码事实。
- 但单个服务 `README` **不能单独推出项目阶段已完成**。

例如：

- 若某服务 `README` 写“Stage 7 当前范围已验收”，而 `当前开发计划` 写“Stage 7 仍缺压测归档”，则应判定为：
  - 模块 README 过期
  - 当前项目正式结论以 `当前开发计划` 为准
  - 本轮先修文档，不得基于 README 把阶段状态写成已完成

## 5. 冲突处理流程

只要出现 README、Prompt、阶段文档、专题规范之间的冲突，必须按下面流程处理。

### 第 1 步：先判定冲突陈述属于哪类信息

必须先回答：当前冲突说的是哪类内容？

- 架构边界？
- 阶段定义？
- 当前阶段状态？
- API？
- WebSocket？
- 数据库？
- 事件？
- 状态机？
- 测试 / 完成判定？
- 只是 README 摘要？

### 第 2 步：回到对应真源文档

不得直接在冲突文档之间“投票”。必须回到第 4 节对应的真源。

### 第 3 步：区分“真冲突”还是“衍生文档过期”

- **若高层真源一致，低层摘要文档冲突**：判定为衍生文档过期，直接修摘要文档。
- **若两个正式真源互相冲突**：判定为真冲突，任务状态设为 `阻塞`，先输出冲突矩阵，再等拍板或统一文档。

### 第 4 步：在同一轮任务里完成最小同步

处理完冲突后，本轮至少要同步：

- 直接被判定为过期的文档
- 与用户当前任务强相关的摘要文档
- 若阶段状态或下一步顺序变化，必须同步 `当前开发计划`

### 第 5 步：未完成同步，不得宣布完成

以下情况一律不得标记为完成：

- 代码改了，但正式文档没改
- 正式文档改了，摘要文档仍保留现行冲突口径
- 已发现冲突，但没有输出冲突矩阵和修正建议

## 6. 文档职责边界（直接用于修库）

### 6.1 根 README

根 README 只负责：

- 项目定位
- 当前能力摘要
- 关键文档入口
- 快速导航

根 README 不负责：

- 定义正式契约
- 单独定义阶段验收结论
- 与 `当前开发计划` 并行维护另一套“当前状态”口径

根 README 若写当前状态，必须附带固定说明：

> 本文档仅作摘要与导航。当前正式阶段、已完成/未完成范围、阻塞项与下一步安排以 `docs/setup/当前开发计划.md` 为准；架构、数据库、接口、事件、状态机与安全等正式约束以对应规范为准。

### 6.2 服务 README

服务 README 只负责：

- 模块职责
- 包结构
- 关键调用链
- 当前模块事实
- 运行与测试入口
- 已知限制

服务 README 不负责：

- 给出项目级阶段结论
- 覆盖 `当前开发计划` 的当前阶段状态
- 覆盖正式接口 / 数据库 / 事件契约

服务 README 若需要提到阶段，只能这样写：

> 本 README 中涉及的阶段表述仅用于说明模块事实或历史上下文；当前项目正式阶段结论以 `docs/setup/当前开发计划.md` 为准。

### 6.3 当前开发计划

当前开发计划只负责：

- 当前正式阶段
- 已完成 / 未完成 / 阻塞项
- 当前实施顺序
- 当前分支基线
- 下一步建议

当前开发计划不负责：

- 改写架构边界
- 改写 API / DB / Kafka / 状态机正式契约
- 代替专题规范给出字段级定义

### 6.4 开发启动手册

开发启动手册只负责：

- 后续实施顺序
- 阶段定义
- 最小产出
- 最小验收基线
- 环境前置条件

开发启动手册不负责：

- 维护每轮最新进展流水
- 代替 `当前开发计划` 持续维护实时阶段状态

### 6.5 Prompt 文档

Prompt 只负责：

- 角色设定
- 执行方式
- 输出结构
- 本任务特有的非目标、红线和交付要求

Prompt 不负责：

- 定义正式契约
- 定义项目正式阶段状态
- 覆盖正式文档

每个 Prompt 必须包含固定限制：

> 本 Prompt 只定义执行方式，不定义正式契约；若与正式规范冲突，以仓库正式规范为准。

## 7. 最小阅读集

AI 不需要机械通读全仓，但也不能绕过与本轮任务直接相关的真源文档。

| 任务类型 | 必读最小集合 | 额外补读 |
| --- | --- | --- |
| 纯文档 / 提示规范 / README 修订 | `AGENTS.md`、本文件、`开发启动手册`、`当前开发计划`、目标文档 | 若内容触及 API / DB / 事件 / 状态机，再补读对应专题规范 |
| 单服务内部实现 / 局部修复 | `AGENTS.md`、`SKILLS.md`、本文件、架构方案、开发启动手册、编码与测试规范、日志规范、完成定义、目标服务 README / 代码 | 若触及缓存、事件、状态迁移、交易时间，再补读对应专题规范 |
| API / DB / 事件 / 状态机修改 | 在“单服务内部实现”基础上，再读相关 API / DB / Kafka / 状态机 / 事务规范与统一接口文档 | 若涉及跨服务共享 JSON，再补读消费方与 contract 模块 |
| 问题清单 / 审计整改 | `AGENTS.md`、本文件、架构方案、开发启动手册、当前开发计划、统一问题清单、目标专题规范 | 必须先核对问题是否真实存在，再决定是否改代码 |
| 阶段收口 / 阶段判断 | `AGENTS.md`、本文件、开发启动手册、当前开发计划、完成定义、目标阶段涉及的专题规范与代表性测试 | 若要写“已完成”，必须补代码证据、测试结论和文档同步结论 |
| Prompt 编写 / 修订 | `AGENTS.md`、本文件、开发启动手册、当前开发计划、目标 Prompt 涉及的专题规范 | 若 Prompt 触及登录、WebSocket、下单、入金、状态机等细节，必须补读对应正式规范 |

## 8. 默认可自主实施范围

在不改变正式契约的前提下，AI 默认可以直接实施下面事项，无需重复确认：

- 同一 owner 服务内的内部实现细节调整
- 私有方法拆分、类内重构、同包内受控抽象
- 注释、日志、测试补齐
- 文档澄清、结构优化、README 与 Prompt 去漂移
- 不改变契约的命名收敛、重复逻辑消重、SQL fragment 抽取
- 与改动范围相称的验证方案选择
- 已确认真实存在的问题，在不触及契约变更时批量修复

原则：

- 用户授权范围内，不要因内部实现细节反复追问。
- 但只要触及正式契约，不能因为“这是实现细节”而绕过确认边界。

## 9. 必须先确认的边界

以下事项属于强确认边界，不能自动拍板：

- API 路径 / 方法 / 字段 / 响应结构变更
- WebSocket 握手、订阅、推送结构、连接限制变更
- 数据库 schema / 索引 / migration 策略变更
- Kafka Topic、payload、分区键、消费语义变更
- 错误码、状态码、状态机迁移规则变更
- 服务边界、owner 边界、包结构边界变更
- 新增跨服务共享模型、共享快照或共享契约
- 项目正式阶段结论、上线口径、生产可用表述变更
- 根 README / 服务 README 中任何会影响“当前正式状态”的表述更新

如果用户已经明确授权这类变更，则无需二次确认；否则必须先确认。

## 10. 验证深度规则

验证深度必须与改动范围相称，不允许默认把所有任务升级成全仓重回归。

| 改动范围 | 默认验证深度 |
| --- | --- |
| 纯文档 / README / Prompt / 计划修订 | 文档一致性检查、链接核对、必要时全文检索交叉核验 |
| 单服务内部逻辑 | 目标模块相关测试、必要的编译验证 |
| 跨服务事件 / 缓存 / 共享 JSON | 生产者与消费者至少各做一轮验证，必要时补链路验证 |
| 数据库 schema / migration | 隔离测试库、目标模块集成测试、checksum 影响核对 |
| 阶段完成 / 真实能力声明 | 必须同时说明代码证据、验证命令、业务链路结论和剩余缺口 |

补充要求：

- `mvn test` 与 `mvn clean compile` 等共享 `target/` 的命令必须串行执行。
- 纯文档任务默认不要求跑全仓测试；若未执行代码验证，必须明确说明。
- 写“阶段已完成”时，不能只写命令名，必须说明对应业务链路结论。

## 11. 文档同步规则（关闭漂移的硬要求）

### 11.1 改正式规则，先改真源，再改摘要

如果本轮改的是正式规则，必须按下面顺序处理：

1. 先改对应真源文档
2. 再改统一接口文档 / 当前开发计划 / README / Prompt 等衍生文档
3. 最后给出验证与影响范围

### 11.2 改当前阶段状态，必须同步状态文档

只要本轮会影响：

- 当前正式阶段
- 已完成 / 未完成
- 当前阻塞项
- 下一步顺序
- 当前分支基线

就必须同步：

- `docs/setup/当前开发计划.md`

必要时再同步：

- 根 `README.md`
- 受影响服务 `README.md`
- 场景 Prompt 摘要

### 11.3 接口 / 事件 / 数据库任务的联动同步

- REST / WebSocket 变更：必须同步对应正式规范和 `FalconX统一接口文档.md`
- Kafka 变更：必须同步 `Kafka事件规范.md`
- 数据库变更：必须同步数据库设计、SQL 蓝图、运行时 migration
- 阶段结论变更：必须同步 `当前开发计划.md`

### 11.4 未同步不闭环

任一任务如果存在下列情况，不得标记完成：

- 真源文档已改，但摘要文档仍保留相反的现行口径
- 当前阶段结论已变，但 `当前开发计划` 未同步
- 接口 / 事件 / 数据库已变，但对应正式规范未同步
- 已发现冲突，但没有修复也没有显式挂阻塞

---

## 12. Prompt 编写规则

后续仓库中的 Prompt 统一按下面原则编写：

1. **Prompt 只定义执行方式，不定义正式契约。**
2. **Prompt 只写本任务特有内容，不全文复制仓库规则。**
3. **Prompt 必须区分：硬要求 / 默认建议 / 可裁剪项。**
4. **Prompt 不能把 README 或当前开发计划当作正式契约来源。**
5. **Prompt 涉及阶段状态时，只能引用 `当前开发计划`。**
6. **Prompt 涉及接口 / 数据库 / 事件 / 状态机时，必须回链到对应正式规范。**

推荐结构：

1. 角色定位
2. 任务目标
3. 适用范围 / 非目标
4. 必读最小集合
5. 交付物
6. 红线约束
7. 验收与验证

---

## 13. 任务输出时的冲突矩阵模板

当发现冲突时，统一按下面格式输出：

| 主题 | 冲突文档 A | 冲突文档 B | 当前冲突点 | 真源判定 | 处理动作 |
| --- | --- | --- | --- | --- | --- |
| 示例：当前正式阶段 | `服务 README` | `当前开发计划` | README 写已完成，计划写未完成 | 以 `当前开发计划` 为准 | 修 README；本轮不改阶段结论 |

若两个正式真源互相冲突，再追加：

- 当前状态：`阻塞`
- 需要拍板：列出最小决策集
- 在拍板前：不得改代码、不得改阶段结论、不得宣布完成

## 14. 与现有文档的关系

- `AGENTS.md / SKILLS.md / 完成定义` 继续承担仓库级红线与完成判定。
- `开发启动手册` 继续负责阶段定义、实施顺序和最小验收基线。
- `当前开发计划` 继续负责“当前时点”的正式阶段状态、阻塞项和下一步。
- `docs/design/codex-prompt.md` 等 Prompt 文档继续只承担场景执行提示，不再承担仓库主规则职责。
- 根 `README.md` 与服务 `README.md` 继续作为摘要文档存在，但不得再与正式规范并行维护另一套现行规则。

## 15. 一句话执行原则

**先判断信息类型，再回真源；真源不一致先阻塞，真源一致而摘要冲突就改摘要；任何 Prompt、README、阶段总结都不能反向改写正式契约。**
