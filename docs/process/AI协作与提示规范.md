# FalconX AI 协作与提示规范

## 1. 目的

本文件用于把 FalconX 仓库里的 AI 协作规则拆成 3 层：

1. **硬约束**：架构、owner、接口、数据库、事件、安全和交付红线，不能突破。
2. **默认策略**：在不触碰硬约束时，AI 应如何高效推进，避免过度阅读、过度确认和过度交付。
3. **场景提示模板**：前端设计、问题审计、后端开发等场景下，提示词应该怎样组织，既不丢边界，也不过度束缚。

本文件不替代 `AGENTS.md / CLAUDE.md`，而是补充“如何在这些强约束下更顺畅地协作”。

## 2. 规则层级

发生冲突时，优先级固定如下：

1. 主架构文档与开发启动手册
2. 数据库 / API / 安全 / Kafka / 状态机 / 事务规范
3. 编码 / 日志 / 测试 / 完成定义
4. `AGENTS.md / CLAUDE.md`
5. 本文件
6. 场景 Prompt（例如前端设计 Prompt）
7. 当前开发计划

结论：

- Prompt 不能覆盖正式契约。
- 当前开发计划可以指导优先级，但不能改写架构边界。
- AI 协作优化只放宽执行方式，不放宽正式边界。

## 3. 任务类型与最小阅读集

AI 不应机械地“每次把所有文档全读一遍”，也不允许跳过与当前任务直接相关的正式规范。默认按下面的最小阅读集执行。

| 任务类型 | 必读最小集合 | 额外按需补读 |
| --- | --- | --- |
| 纯文档 / 提示规范 / 计划优化 | 架构文档、开发启动手册、当前开发计划、统一问题清单、目标文档 | 若文档内容触达 API / DB / 事件 / 状态机，再补读对应专题规范 |
| 单服务内部实现 / 局部修复 | 架构文档、开发启动手册、编码测试规范、日志规范、完成定义、目标服务 README / 相关代码 | 若涉及缓存、事件、交易时间、状态迁移，再补读对应专题规范 |
| API / 事件 / 数据库 / 状态机修改 | 在“单服务内部实现”基础上，再读相关 API / DB / Kafka / 状态机 / 事务规范和统一接口文档 | 若涉及跨服务共享 JSON，再补读消费者侧契约 |
| 问题清单 / 审计整改 | 架构文档、开发启动手册、统一问题清单、目标问题涉及的专题规范 | 必须先核对问题是否真实存在，再决定是否改代码 |
| 前端产品设计 / 原型 | 架构文档、开发启动手册、当前开发计划、统一接口文档、目标设计文档 | 若触达登录、WebSocket、状态机、入金、下单等细节，再补读 REST / WS / 安全 / 状态机 / 市场契约 |

补充要求：

- 任务只要不触达某一专题，就不要求为了“形式完整”去机械通读该专题。
- 任务一旦触达某一专题，就不能只看二手摘要，必须回到正式规范。

## 4. AI 可自主决策范围

在不改变正式契约的前提下，AI 默认可以直接实施下面这些事项，无需重复向用户确认：

- 同一 owner 服务内的内部实现细节调整
- 私有方法拆分、类内重构、同包内受控抽象
- 注释、日志、测试补齐
- 文档澄清、结构优化、提示词模板优化
- 不改变契约的命名收敛、重复逻辑消重、SQL fragment 抽取
- 与改动范围相称的验证方案选择
- 问题清单中“已确认真实存在”的问题，在不触及契约变更时批量修复

这里的“可自主决策”包括但不限于：

- 算法选择
- 私有类 / 私有方法拆分
- 同 owner 内的目录整理
- 测试用例组织方式
- 注释粒度
- 日志字段组装方式

原则：

- 同一轮任务里，用户已经授权的范围内，不要因为内部实现细节反复追问。
- 若正式规范已经冻结，AI 应直接执行，不要把已冻结事项再次抛回给用户。

## 5. 必须先确认的范围

以下事项仍然属于强确认边界，不能因为“AI 协作优化”而放宽：

- API 路径 / 方法 / 字段 / 响应结构变更
- 数据库 schema / 索引 / migration 策略变更
- Kafka Topic、payload、分区键、消费语义变更
- 错误码、状态码、状态机迁移规则变更
- 服务边界、owner 边界、包结构边界变更
- 新增跨服务共享模型、共享快照或共享契约
- 对外承诺的阶段完成状态、上线口径或生产可用表述变更

如果用户已经明确授权这类变更，则无需二次确认；否则必须先确认。

## 6. 验证深度规则

验证应与改动范围相称，不应默认把所有任务都升级成全仓重回归。

| 改动范围 | 默认验证深度 |
| --- | --- |
| 纯文档 / 提示词 / 计划修订 | 文档一致性检查、链接与表述核对、必要时 `rg` 交叉核验 |
| 单服务内部逻辑 | 目标模块相关测试、必要的编译验证 |
| 跨服务事件 / 缓存 / 共享 JSON | 生产者与消费者两侧至少各做一轮验证，必要时补链路验证 |
| 数据库 schema / migration | 隔离测试库、目标模块集成测试、checksum 影响核对 |
| 阶段完成 / 真实能力声明 | 必须说明代码证据、验证命令和剩余缺口，不能只写“已完成” |

补充要求：

- `mvn test`、`mvn clean compile` 这类共享 `target/` 的命令仍必须串行执行。
- 纯文档任务默认不要求跑全仓测试；若无法验证，必须明确说明。

## 7. 提示词编写原则

后续仓库中的提示词文档，统一按下面方式写：

1. **全局规则放在仓库规则里**
   `AGENTS.md / CLAUDE.md` 负责红线和交付要求，不在每个场景 Prompt 里全文重复。

2. **场景 Prompt 只补充该场景特有内容**
   例如前端设计 Prompt 重点补充用户旅程、视觉方向、页面清单、前后端契约摘要。

3. **Prompt 必须区分三种语气**
   - 硬要求：不满足即视为越界或未完成
   - 默认建议：无更强用户指令时采用
   - 可裁剪项：当用户只要局部交付时允许缩减

4. **避免把“完整大方案”强加给局部任务**
   - 用户要完整方案，才要求全量 PRD + 原型 + Token + 落地建议
   - 用户只要单页、组件、评审或修订时，不强行扩展到全站交付

5. **避免重复复制正式契约**
   - 关键契约可以摘要
   - 完整口径应回链到正式文档

6. **避免不必要的长篇复述**
   - 不要求每次读完文档后都写长篇总结
   - 用简短“关键约束摘要”即可证明已理解上下文

## 8. 推荐 Prompt 结构

任一场景 Prompt，推荐按下面 7 段组织：

1. 角色定位
2. 任务目标
3. 适用范围 / 非目标
4. 必读最小集合
5. 交付物
6. 红线约束
7. 验收与验证

这样做的目的：

- 保留边界
- 降低重复
- 允许局部任务裁剪
- 让 AI 在不同任务之间复用同一套执行方式

## 9. 与现有文档的关系

- `docs/design/codex-prompt.md` 现在应被理解为“前端产品设计场景 Prompt”，不是全仓库的主 Prompt。
- `AGENTS.md / CLAUDE.md` 继续承担全仓库硬约束。
- 当前开发计划继续承担进度和优先级说明，不承担正式契约定义。
